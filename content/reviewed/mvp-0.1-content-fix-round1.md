## 1. 本轮结论

12 讲诊断题不需要推倒重写，主要问题是答案位置失衡、弱干扰项和解析未充分说明“为什么其他项不是第一步”。本轮给出 12 题逐题修订对照，保持现有 schema 结构，单选只通过重排选项、改写选项文案和增强解析来打散答案分布。

修订后 11 道单选正确答案分布为：

| 选项 | 数量 | 占比 |
|---|---:|---:|
| A | 3 | 27.3% |
| B | 2 | 18.2% |
| C | 3 | 27.3% |
| D | 3 | 27.3% |

结论：A/B/C/D 均出现，任一选项占比不超过 40%。12 道诊断题中 10 道具备“看似合理但优先级不对”的强干扰项，占比 83.3%，超过 30% 门槛。

## 5. 去模板化写作约束

后续 44 讲可复制字段结构和章节顺序，但不能复制固定条数、固定句式和固定题目套路。

正文约束：

- `mechanism` 写 4-7 条，按概念复杂度和因果链自然决定；不要为了凑 6 条拆弱句。
- `pitfalls` 写 3-6 条，必须是真实工程误判；不要为了凑 5 条写泛泛提醒。
- `keyTakeaways` 写 3-5 条，只保留学习者能带走的判断；不要每讲固定 5 条。
- `mentalModel` 不固定“可以把 X 理解为……”句式。可用类比、反例、边界条件、工程角色视角、错误用法对照等方式表达。
- `enterpriseCase` 必须包含现象、问题、分析、方案和结论；至少补 2 类工程信号：指标、规模、系统边界、错误路径、约束条件或验证结果。
- `relatedConceptIds` 只放强相关概念，弱关系放标签或术语，不要把能联想到的概念都列进去。

心智模型可替换句式示例：

- “在工程上，它更像……”
- “不要把它当成……，它真正约束的是……”
- “反过来看，如果没有它，系统会先在哪个环节失控……”
- “从平台负责人的视角，它是……”
- “它解决的不是……，而是……”

## 7. 动画画面意图初稿

以下只给动画语义和画面意图，不写前端实现，不要求修改 `AnimationConfig` 协议。

### 7.1 `prefill-decode` 画面意图

总体画布语义：

- 画面主体是一条从“请求进入”到“首个 Token”再到“后续 Token 流出”的横向时间轴。
- 时间轴分为首字前和首字后两段：首字前包括网关、RAG/工具、Prefill、KV 写入；首字后是 Decode 循环。
- TTFT 用从请求起点到首个 Token 标记的长度表达，TPOT 用相邻输出 Token 的间隔表达。
- `highlightTargets` 不作为英文标签展示，而映射为画面元素的高亮、拉长、变暗或路径变化。

用于 Prefill 的画面语义：

| step | 画面意图 |
|---|---|
| 组装完整上下文 | `prompt`、`rag-chunks`、`history` 显示为三类输入块，汇入同一输入序列；无关或过长块可以用偏灰色显示。 |
| 进入 Prefill | `input-tokens` 变成一条可伸缩 Token 条，进入 `prefill` 区；Token 条越长，首字前时间轴越长。 |
| 写入 KV Cache | `kv-write` 表现为 Key/Value 小格写入 `cache` 区，说明 Prefill 不只是读输入，也在准备后续复用状态。 |
| 首个 Token 准备输出 | `first-token` 在时间轴上亮起，`ttft` 标尺停止，提示首字前阶段结束。 |
| 长上下文拖慢首字 | `long-context` 拉长输入条，`ttft-growth` 让首字前时间段明显变长，对比短上下文状态。 |

用于 Decode 的画面语义：

| step | 画面意图 |
|---|---|
| Prefill 完成 | `prefill-done` 区域变为完成态，`cache` 区保持亮起，表示 Decode 已有可用上下文状态。 |
| 生成第一个输出 Token | `first-output-token` 从 Decode 区冒出，用户侧开始看到响应。 |
| 追加并继续预测 | `append-token` 把新 Token 接到输出序列末尾，`decode-loop` 用小循环箭头表示继续预测下一 Token。 |
| TPOT 体现流出速度 | `tpot` 显示为相邻 Token 间距标尺，`token-interval` 变宽时表示输出卡顿。 |
| 长输出累积总耗时 | `long-output` 拉长输出序列，`total-latency` 增长，但首个 Token 位置不变，强调长输出主要影响总耗时。 |

用于 TTFT 的画面语义：

| step | 画面意图 |
|---|---|
| 请求进入网关 | `gateway` 是入口节点，`queue` 显示为排队条，`route` 显示为从网关到模型池的路径选择。 |
| RAG 与工具前置处理 | `rag` 和 `tool-call` 是模型前的两个耗时块，任一块变长都会推迟首个 Token。 |
| Prefill 处理输入 | `prefill` 区块接收 `input-tokens`，根据输入长度占据不同时间宽度。 |
| 首个 Token 返回 | `first-token` 沿 `stream` 返回到客户端，TTFT 标尺停在这一点。 |
| 拆分瓶颈归因 | `latency-breakdown` 展示分段条，网关、RAG、Prefill、网络各自占用不同颜色或纹理。 |

### 7.2 `agent-loop` 画面意图

总体画布语义：

- 画面主体是一个环形循环：Goal/Constraints/Tools -> Observe -> Plan -> Act -> Check -> Decision。
- Decision 节点分三条路径：继续循环、停止完成、转人工审批。
- 下方保留一条 trace 轨道，记录每轮观察、动作、结果和继续条件，强调可审计。
- 高风险动作从 Act 或 Check 分支进入 Human Review，而不是简单继续下一轮。

逐步画面语义：

| step | 画面意图 |
|---|---|
| 用户给出目标 | `goal` 在环形左上方出现，`constraints` 固定为边界栏，`tools` 显示为可用工具列表但带权限边界。 |
| Observe 观察 | `observe` 节点亮起，从 `context`、日志或文件区拉取证据卡片进入当前状态。 |
| Plan 选择下一步 | `plan` 节点将证据卡片转成一个可验证动作，不展示长推理文本，只展示“下一步动作”和“验证标准”。 |
| Act 调用工具 | `act` 节点向工具区发出调用箭头，`tool-result` 返回成功或错误状态，并写入 trace 轨道。 |
| Check 校验结果 | `check` 节点对照目标和约束，`evidence` 卡片显示测试、日志、diff 或人工反馈证据。 |
| 继续、停止或转人工 | `continue` 回到 Observe，`stop` 进入完成态，`human-review` 走到人工审批分支；如果连续失败，继续箭头变弱并强制转人工。 |

动画审核注意：

- 不直接展示 `goal`、`human-review` 等 raw key 文本；这些 key 只作为画布元素映射依据。
- 画面重点是循环、证据、退出和人审分支，不是把步骤说明排成列表。
- reduced-motion 下应仍能看到环形结构、当前步骤、trace 轨道和三种退出路径。
