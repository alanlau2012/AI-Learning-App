# capability-routing：能力路由

## 一句话定义
能力路由是根据任务需要的推理、代码、工具、长上下文、多模态或合规能力，把请求分配给真正胜任的模型或链路。

## 为什么重要
企业平台不能只按模型大小选路由。一个便宜模型可能足够做摘要，却不适合多文件代码修改；一个强模型可能会浪费在简单分类上。能力路由让平台按任务需求选择能力，而不是按品牌或默认配置选择模型。

## 心智模型
能力路由像派工单。修水管、审合同和写代码需要不同技工；派最贵的人做所有事浪费，派错人做关键事会返工。

## 机制拆解
1. 请求进入平台后先识别任务类型、输入形态、工具需求和风险等级。
2. 模型或链路需要维护能力标签，例如长上下文、代码、结构化输出、工具调用和合规拒答。
3. 路由策略把任务需求与能力标签匹配，过滤不达标候选。
4. 对复杂任务保留升级、重试或人工审核路径。
5. 评测集和线上 trace 用于修正能力标签，避免凭经验维护路由。

## 动画脚本或无动画说明
本讲复用 `model-router`。画面重点是请求标签、候选模型能力画像、过滤不达标模型、选中合适模型和评测/观测回流。

## 企业案例
标题：代码修复任务被路由到摘要模型。场景：研发平台同时提供会议摘要、知识问答和代码修复能力。问题：15% 代码修复请求被路由到通用摘要模型，PR 验证失败率达到 38%。分析：路由只看成本和平均延迟，没有识别仓库上下文、工具调用和测试能力需求。方案：新增任务识别、能力标签和代码任务评测集，代码修复默认走具备 repo-context 和工具调用能力的链路，失败时升级人工。结论：能力路由要用任务能力要求约束模型选择。工程信号：指标、系统边界、错误路径、验证结果、约束。

## 常见误区
1. 认为大模型能力最强，所以可以覆盖所有任务。
2. 只按成本或延迟路由，不看任务需要的能力。
3. 能力标签靠人工印象维护，没有评测和 trace 回流。
4. 忽略工具、上下文和结构化输出能力，把它们都归为模型能力。
5. 没有不确定时的升级和人工审核策略。

## 诊断题
场景：平台把合同审查、FAQ 和代码修复都按成本优先路由。最近代码修复失败率升到 38%，日志显示很多请求没有加载仓库上下文和测试工具。问题：最应该补哪项能力？A. 把所有任务都切到旗舰模型。B. 把低价模型并发调高。C. 建立任务识别与能力标签，让代码修复走支持仓库上下文和工具调用的链路。D. 缩短系统提示词，降低首字时间。正确答案：C。解析：C 对准任务能力不匹配；A 是强干扰项，可能缓解部分问题但仍可能缺工具链；B/D 优化性能，不解决能力路由。排查路径：按任务类型统计失败；检查路由标签；补能力矩阵；回放代码任务评测；灰度新策略并监控验证失败率。

## 核心结论
1. 能力路由按任务需求选链路，不按模型名选链路。
2. 能力包括模型、工具、上下文、格式和治理约束。
3. 能力标签必须由评测和 trace 持续校准。
4. 复杂任务需要升级和人审兜底。
