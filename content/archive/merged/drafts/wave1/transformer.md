# transformer：Transformer

## 一句话定义
Transformer 是现代大模型的核心神经网络架构，它通过注意力机制和多层表示变换，把上下文中的 token 关系逐层加工成可用于预测下一个 token 的内部状态。

## 为什么重要
Transformer 决定了大模型为什么能在同一套架构中处理摘要、问答、代码、规划和工具调用等任务。对企业工程负责人来说，理解 Transformer 不是为了手写模型，而是为了判断上下文长度、注意力开销、推理延迟、RAG 证据组织、长文本效果衰减和模型能力边界。当系统出现首字延迟高、长上下文质量下降、证据被忽略或复杂指令不稳定时，很多现象都能追溯到 Transformer 如何在有限计算中分配上下文关系。

## 心智模型
把 Transformer 看成一座多层加工厂：原料是 token 序列，每一层都会重新评估“哪些位置与当前预测有关”，再把局部线索加工成更抽象的表示。工厂可以处理很多任务，但每次加工都受输入长度、位置编码、注意力计算和模型参数的限制；给它更多材料不一定更好，材料组织错误反而会占用加工能力。

## 机制拆解
1. 输入 token 会先变成 embedding，并叠加位置信息，让模型同时知道“是什么”和“在序列哪里”。
2. 每一层通过自注意力计算 token 之间的关系，使当前位置能够利用历史上下文中的相关信息。
3. 前馈网络会对注意力聚合后的表示做非线性变换，提炼更适合预测的特征。
4. 多头注意力允许模型从不同关系视角处理上下文，例如指代、格式、约束、证据和语法。
5. 层数、隐藏维度、注意力头数和上下文长度共同影响能力、显存、吞吐和延迟。
6. 自回归大模型在推理时基于当前上下文表示预测下一个 token，并把新 token 追加回上下文继续计算。

## 企业案例
**标题：长制度问答在 20 页附件后开始忽略关键约束**

- 场景：某集团法务助手允许用户上传合同和内部制度，单次请求平均 35k 输入 token，峰值超过 90k。
- 问题：模型能流畅回答，但对“例外条款”和“审批层级”漏读率高；抽样 300 个问题，长上下文下关键约束命中率从 86% 降到 63%。
- 分析：问题不是单纯“模型不够聪明”。长输入让注意力计算和证据竞争变重，RAG 片段、历史对话和附件全文混在一起，关键条款没有被结构化置顶；平台只监控总 token，没有监控证据命中和引用位置。
- 方案：把全文直塞改为“检索候选 + 条款重排 + 关键约束置顶 + 引用校验”；限制附件全文进入上下文的比例，并建立长上下文回放集。
- 结论：Transformer 能利用上下文，但企业系统要主动组织上下文，而不是把所有材料交给模型自行消化。

## 常见误区
1. 把 Transformer 理解成“万能理解器”，忽略输入组织和计算预算。
2. 认为上下文越长越安全，实际可能让关键证据在注意力竞争中被稀释。
3. 只比较模型榜单分数，不比较自己任务上的延迟、引用准确率和失败样本。
4. 把所有问题都归因于参数量，不检查 RAG、提示词、位置和分片。
5. 忽略推理阶段的 KV Cache、Prefill 和 Decode 对性能的影响。

## 诊断题
**场景**：企业知识库助手换成长上下文模型后，用户可以一次上传更多材料，但复杂制度问答的准确率没有提升，TTFT 从 1.2s 上升到 5.8s，错误样本多为忽略附件中后段的限制条款。

**问题**：最优先的判断是什么？

A. 长上下文模型不适合企业问答，应立即退回短上下文模型  
B. 检查上下文组织、证据排序和关键约束位置，而不是只看窗口变大  
C. 把 temperature 调低到 0，优先消除生成随机性  
D. 增加更多附件全文，让模型拥有更完整的信息

**正确答案**：B

**解析**：B 对应 Transformer 在有限计算中处理上下文关系的工程问题。A 过早下结论，短窗口未必更好。C 可能降低随机性，但不能解决证据位置和注意力竞争。D 是强干扰项，看似补充信息，实际可能继续提高 Prefill 成本并加剧噪声。

**排查路径**：
1. 对错误样本标注正确证据所在位置。
2. 检查 RAG 片段排序、附件截断和关键条款是否置顶。
3. 分别回放全文直塞、检索片段、重排片段三种输入。
4. 记录 TTFT、输入 token、引用准确率和约束命中率。
5. 用失败样本校准上下文拼接策略。

## 核心结论
1. Transformer 的能力来自多层注意力和表示变换，而不是神秘理解。
2. 长上下文提高容量，同时提高成本和证据竞争。
3. 企业系统要设计上下文组织策略，让模型看到正确材料。
4. 评估 Transformer 相关方案时要同时看质量、延迟、成本和失败路径。
5. RAG、Agent、MaaS 性能治理都建立在对 Transformer 推理特性的理解之上。
