# autoregressive：自回归生成

## 一句话定义
自回归生成是大模型每次基于已有上下文预测下一个 token，再把新 token 接回上下文继续生成的机制。

## 为什么重要
自回归解释了为什么输出是逐字出现、为什么长答案更慢、为什么错误会沿着后续文本扩散，也解释了 Agent 为什么需要中途检查和停止条件。企业系统里，Decode 延迟、输出阶段单 token 间隔、最大输出长度、流式响应、工具调用循环和幻觉扩散都与自回归有关。排查时不能只看总耗时，要拆开首字前和首字后的生成链路。

## 心智模型
模型不是一次性把完整答案吐出来，而像一名边写边读自己草稿的助手。每写下一个 token，它都会成为下一步判断的一部分；前面一旦选错方向，后面的文字往往会继续沿着这个方向补全，看起来越来越“自洽”，但不一定越来越正确。

## 机制拆解
1. 推理开始时，模型先根据输入上下文完成 Prefill，得到可用于生成的内部状态。
2. Decode 阶段每一步都会根据当前上下文分布预测下一个 token。
3. 被选中的 token 会追加到上下文中，参与下一步预测。
4. 生成长度越长，Decode 循环次数越多，总耗时和输出成本越高。
5. 如果早期 token 偏离事实或格式，后续 token 可能沿着错误上下文继续扩写，因此需要引用校验、停止条件和中途检查。

## 动画脚本或无动画说明
本讲复用 `token-flow` 动画，不新增动画组件或协议。画面聚焦 token-flow 的后半段：`decode`、`output-tokens`、`cost` 表示逐 token 生成，`input-text`、`tokens`、`token-ids` 作为前置状态。highlightTargets 可映射到已有 TokenFlowAnimation 可视元素：

1. `input-text`：显示原始输入进入链路。
2. `tokens`：提示上下文已被拆成 token 序列。
3. `prefill`：显示首个输出前的上下文处理。
4. `decode`, `output-tokens`：突出每次只生成一个新 token。
5. `cost`：表示输出越长，循环次数和成本越高。

## 企业案例
**标题：客服助手长回答导致尾部错误扩散**

- 场景：某客服助手每天约 12 万次对话，开启流式输出和“详细解释”模式后，平均输出从 180 token 增加到 620 token。
- 问题：用户满意度短期上升，但抽检发现后半段补充说明的事实错误率从 4% 上升到 13%，单 token 输出间隔稳定但总输出耗时明显变长。
- 分析：模型在前半段给出正确方向后，继续自回归扩写，尾部缺少检索证据支持；系统只限制输入 token，没有限制答案结构和停止条件。
- 方案：把回答拆成“直接结论、依据、下一步动作”三段，设置最大输出 token 和引用覆盖要求；对无证据扩展触发“建议转人工/补充检索”。
- 结论：自回归生成越长，越需要结构、证据和停止条件控制。

## 常见误区
1. 认为流式输出变快等于总生成变快；流式只改善感知首字，不减少后续 Decode 次数。
2. 用“回答更详细”掩盖无证据扩写，导致尾部幻觉增多。
3. 只调 temperature，不限制输出结构、证据覆盖和停止条件。
4. 把首字慢和全文慢混在一起排查，忽略 Prefill 与 Decode 分工。

## 诊断题
**场景**：客服助手开启“详细模式”后，首字时间几乎不变，但完整回答耗时翻倍，后半段事实错误明显增多。监控显示平均输出 token 增长 3 倍。

**问题**：最应该优先采取哪项措施？

A. 扩容推理实例，先提高整体吞吐  
B. 把 temperature 调到 0，减少随机性  
C. 增加更多历史对话，帮助模型保持一致  
D. 限制输出结构和最大生成长度，并对无证据扩写设置停止或补检索

**正确答案**：D

**解析**：D 直接对应自回归 Decode 变长和错误扩散。A 是强干扰项，扩容能缓解排队但不能减少单次输出循环和幻觉尾巴。B 可能降低随机性，但仍会在错误上下文里稳定扩写。C 可能加重输入负担，也不解决输出过长。

**排查路径**：
1. 拆分首字时间、输出阶段单 token 间隔、输出 token 数和总耗时。
2. 抽检错误出现在答案的哪个段落。
3. 对比详细模式和结构化短回答的事实错误率。
4. 设置最大输出长度、证据引用和停止条件。
5. 回放高风险问题并观察尾部幻觉是否下降。

## 核心结论
1. 自回归生成是一轮一轮预测下一个 token。
2. 输出越长，Decode 循环、成本和错误扩散风险越高。
3. 流式输出改善感知，不等于减少完整生成成本。
4. 企业应用需要用结构、证据和停止条件管理生成过程。
