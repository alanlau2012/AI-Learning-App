# positional-encoding：位置编码

## 一句话定义
位置编码是让模型知道 token 在序列中相对或绝对位置的机制，避免 Transformer 只看到一袋 token 而无法区分顺序、距离和结构。

## 为什么重要
企业应用中，顺序经常就是含义：审批链条、合同条款层级、代码调用顺序、对话轮次和工具执行结果都不能随意打乱。位置编码解释了为什么长上下文模型会有有效长度、为什么靠后的证据可能被忽略、为什么把关键约束放在提示词开头或结尾会影响效果。平台侧要关注的不只是窗口大小，还包括截断策略、上下文拼接顺序、位置外推能力和长文本回放评测。

## 心智模型
同一组材料放进模型，就像把文件摆在一张长桌上。位置编码给每份材料贴上座位号；座位号本身不是内容，但它会影响模型在处理“前文说过什么、当前该看哪里、谁约束谁”时的判断。座位安排混乱，模型可能不是不知道规则，而是没在正确位置上使用规则。

## 机制拆解
1. Transformer 的注意力本身不天然知道 token 顺序，需要额外位置信号参与计算。
2. 位置编码可以是绝对位置、相对位置、旋转位置编码等形式，具体实现会影响长上下文表现。
3. 推理时，token 的位置会随上下文长度增长而变化；同一证据放在不同位置，效果可能不同。
4. 当输入超过模型训练或优化良好的长度区域时，位置外推和注意力稳定性可能下降。

## 动画脚本或无动画说明
本讲为纯文本内容，不新增动画。位置机制可由正文和案例解释，不硬接动画。

## 企业案例
**标题：审批规则放在长上下文末尾后被忽略**

- 场景：某费用报销助手把用户历史对话、报销制度、发票 OCR 和审批规则拼成上下文，平均输入 18k token。
- 问题：当审批规则位于上下文末尾时，模型对“超过 5000 元需二级审批”的命中率只有 70%；把规则前置并压缩历史后，命中率提升到 91%。
- 分析：系统边界没有区分“背景材料”和“硬约束”。位置顺序让模型更容易受近期 OCR 和用户补充说明影响，关键规则虽然在窗口内，但没有被稳定使用。
- 方案：将硬约束固定放在系统提示词后第一段，对历史对话做摘要，对 OCR 结果只保留与金额、票据类型、部门相关的结构化字段。
- 结论：位置不是排版细节，而是上下文工程的一部分；关键约束要有稳定位置和验证用例。

工程信号：指标、系统边界、约束条件、验证结果。

## 常见误区
1. 认为只要内容在上下文窗口内，模型就会等价利用。
2. 把关键约束、用户输入、RAG 片段和工具结果随意拼接。
3. 只测试短输入，不测试长输入和多轮对话后的规则命中率。

## 诊断题
**场景**：报销 Agent 在单轮测试中能正确执行“超过 5000 元二级审批”，但多轮对话后经常漏掉。日志显示规则仍在上下文中，只是排在历史对话和 OCR 明细之后。

**问题**：第一步应该做什么？

A. 把模型换成更大参数版本，提升规则理解能力  
B. 增加一条提示词：“请认真遵守所有规则”  
C. 固定硬约束在上下文中的优先位置，并回放多轮样本验证命中率  
D. 删除所有历史对话，只保留最后一轮用户输入

**正确答案**：C

**解析**：C 直接处理位置编码和上下文组织带来的工程问题。A 可能有帮助，但在规则仍被错误摆放时不是第一步。B 只是弱约束，不能保证关键规则稳定进入有效位置。D 可能降低干扰，但会丢失必要对话信息，属于过度处理。

**排查路径**：
1. 标注失败样本中硬约束的位置。
2. 比较规则前置、后置、重复锚定三种拼接策略。
3. 记录多轮长度、规则命中率和错误类型。
4. 将历史对话压缩为结构化摘要。
5. 为关键规则建立回归测试。

## 核心结论
1. 位置编码让顺序成为模型输入的一部分。
2. 在企业上下文工程中，材料摆放顺序会影响质量。
3. 关键约束要有稳定位置、结构化表达和回放验证。
