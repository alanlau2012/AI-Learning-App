# sla：SLA 保障

## 一句话定义
SLA 保障是把模型平台的可用性、延迟、错误率、质量和降级策略定义为可承诺、可监控、可复盘的服务目标。

## 为什么重要
企业用户不会只问模型能不能回答，而会问关键业务是否稳定可用。SLA 把平台责任从最好努力变成明确承诺：哪些任务要 P95 低于多少、可用性多少、失败怎么降级、质量如何评估、事故如何复盘。

## 心智模型
SLA 像服务合同。它不是一句保证稳定，而是写清楚交付指标、例外情况、补救手段和责任边界。

## 机制拆解
1. 平台按业务等级定义可用性、P95/P99、错误率、超时率和质量底线。
2. 指标必须拆到模型、网关、队列、RAG、工具和外部供应商等链路段。
3. 降级策略包括模型切换、缓存兜底、缩短输出、关闭非关键能力和人工升级。
4. SLA 需要告警、值班、事故分级和复盘机制支撑。
5. 质量类 SLA 依赖 Eval、Trace 和线上抽检，不只看 HTTP 成功率。

## 动画脚本或无动画说明
本讲纯文本，不接动画。它整合模型网关、路由、限流熔断、观测和评测能力。

## 企业案例
标题：合同审核助手 HTTP 成功但业务 SLA 失败。场景：法务平台给 300 名合同经理提供审核助手，承诺工作时间可用性 99.5%、P95 首字 3 秒内、关键条款漏检率低于 2%。问题：HTTP 成功率 99.8%，但 P95 TTFT 多次超过 7 秒，抽检发现关键条款漏检率 5.6%。分析：平台只监控接口成功率，没有把质量和关键路径延迟纳入 SLA，也没有降级到规则库和人工复核。方案：建立链路指标、质量抽检、失败升级和降级策略；高风险合同触发人工复核，普通合同使用缓存和中模型分流。结论：AI SLA 必须同时覆盖技术可用性、体验延迟和业务质量。工程信号：规模、指标、系统边界、约束、验证结果。

## 常见误区
1. 把 SLA 等同于 HTTP 200 成功率。
2. 只承诺平均延迟，不看 P95/P99 和关键任务。
3. 没有质量指标，导致模型回答错误也算成功。
4. 降级策略临时手工处理，没有预案和演练。
5. 不区分业务等级，对所有请求承诺同一指标。

## 诊断题
场景：合同审核助手 HTTP 成功率 99.8%，但法务投诉高风险合同漏检，P95 首字也从 2.5 秒升到 7 秒。当前 SLA 只写接口可用性。问题：最应该补充什么？A. 只把可用性目标从 99.5% 提高到 99.9%。B. 把所有请求都切到旗舰模型。C. 在 SLA 中加入关键链路延迟、质量抽检指标、降级/人工复核策略和事故复盘。D. 关闭 trace，减少日志成本。正确答案：C。解析：C 覆盖体验和质量两个缺口；A 只提高 HTTP 目标，不能解决漏检；B 是强干扰项，可能增慢且不保证质量；D 会削弱排查。排查路径：拆分技术成功率、TTFT/P99 和质量指标；定义业务等级；补质量评测和抽检；设计降级与人工升级；建立事故复盘。

## 核心结论
1. AI SLA 要覆盖可用性、延迟、错误率和业务质量。
2. HTTP 成功不等于用户任务成功。
3. SLA 必须配套降级、告警、值班和复盘。
4. 不同业务等级应有不同承诺和成本策略。
