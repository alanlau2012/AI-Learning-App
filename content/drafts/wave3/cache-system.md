# cache-system：缓存体系

## 一句话定义
缓存体系是在 AI 平台中复用可安全复用的输入、上下文、结果或中间状态，以降低成本、延迟和重复计算。

## 为什么重要
AI 应用的缓存不只是普通 HTTP 缓存。提示词、RAG 片段、语义相似问题、模型结果、前缀和 KV Cache 都可能被复用，但复用错误会产生过期答案、权限泄漏或错误扩散。

## 心智模型
缓存体系像办公室里的资料夹。常用资料放手边能省时间，但如果资料过期、拿错权限或把别人的批注复用到你的报告里，效率就会变成事故。

## 机制拆解
1. 平台先识别可缓存对象：静态提示词、检索结果、语义相似请求、模型输出、前缀或 KV 状态。
2. 不同缓存有不同命中条件：精确匹配、语义相似、用户权限、模型版本、时间窗口和上下文边界。
3. 缓存命中可降低 TTFT、TPOT、Token 成本或下游系统压力。
4. 缓存必须有失效、隔离和审计策略，避免复用过期或越权内容。
5. 命中率、节省 Token、错误复用率和回源成本要一起监控。

本讲纯文本，不接动画。KV Cache 已由 `kv-cache` 与 `session-affinity` 讲覆盖，本讲聚焦平台层多种缓存边界。

## 企业案例
标题：语义缓存命中率高但投诉增加。场景：客服平台为 FAQ 问答启用语义缓存，日均 50 万次请求。问题：缓存命中率达到 42%，模型成本下降 28%，但政策类问题投诉率上升，部分答案引用了旧规则。分析：缓存只按语义相似命中，没有绑定知识库版本、用户地区和政策生效时间。方案：将缓存 key 加入知识库版本、地区、租户和有效期；高风险政策问题改为短 TTL 并强制回源验证。结论：缓存收益必须和正确性、权限和时效一起治理。工程信号：规模、指标、错误路径、约束、验证结果。

## 常见误区
1. 只追求命中率，不看错误复用率。
2. 把所有相似问题都缓存，不绑定权限、版本和时间。
3. 混淆结果缓存、语义缓存、RAG 缓存和 KV Cache。
4. 缓存命中后不保留 trace，无法解释答案来源。
5. 缓存策略不上线灰度，一次影响所有租户。

## 诊断题
场景：语义缓存上线后成本下降 25%，但合规问答出现旧政策答案。缓存命中日志没有记录知识库版本和地区。问题：优先修复什么？A. 把缓存 TTL 调到更长，提高命中率。B. 全部关闭缓存，避免任何风险。C. 把所有问题都改用旗舰模型。D. 将缓存 key 和失效策略绑定知识库版本、租户/地区、权限和有效期，并监控错误复用。正确答案：D。解析：D 处理错误复用根因。A 会放大过期风险；B 可临时止血但不是体系化方案；C 不解决缓存命中复用旧答案。排查路径：定位错误命中的缓存类型；检查 key 组成；补版本和权限隔离；设置风险分级 TTL；监控命中率与错误复用率。

## 核心结论
1. AI 缓存要同时治理命中率、正确性、权限和时效。
2. 不同缓存层解决不同瓶颈，不能混为一谈。
3. 缓存 key 必须纳入模型、知识、租户和风险边界。
4. 高风险任务宁可少命中，也不能错误复用。
