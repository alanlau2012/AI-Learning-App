# tpot：TPOT

## 一句话定义
TPOT 是首个输出 token 之后，每个后续 token 的平均生成耗时，用来衡量 Decode 阶段“说得快不快”。

## 为什么重要
TTFT 决定用户等多久开始看到响应，TPOT 决定响应开始后是否顺滑。客服、代码生成、长文总结和 Agent 日志流中，TPOT 高会表现为文字一卡一卡、完整答案拖很久、流式输出体验差。平台侧要同时看 TPOT、输出 token 数、P50/P95/P99、batch 大小、KV Cache 命中、显存带宽和并发队列，而不能只看总耗时。

## 心智模型
首字像主持人终于开口，TPOT 像他说话的语速。开口慢可能是前面读材料和排队；说话慢多半发生在逐 token Decode 阶段，和模型大小、并发、缓存、batch、显存带宽以及解码策略有关。

## 机制拆解
1. Prefill 完成后，模型进入 Decode，每一步生成一个或少量 token。
2. TPOT 统计首 token 之后后续 token 的平均间隔，主要反映 Decode 阶段效率。
3. 模型越大、并发越高、KV 读写越重、显存带宽越紧，TPOT 越容易上升。
4. Batch 调度会影响 TPOT：大 batch 提升吞吐，但可能让单个请求等待更久或间隔变宽。
5. 投机解码、量化和缓存优化可能降低 TPOT，但收益取决于命中率、质量回归和硬件适配。

## 企业案例
**标题：代码助手首字很快但输出像挤牙膏**

- 场景：某研发平台的代码助手日均 9 万次补全，升级到更大模型后首字时间维持在 1.2 秒左右。
- 问题：开发者反馈响应开始后明显变慢，平均 TPOT 从 38ms 升到 115ms，长函数生成的 P95 完成时间增加 2.4 倍。
- 分析：Prefill 优化和缓存仍有效，所以 TTFT 没有恶化；真正瓶颈在 Decode 阶段，模型更大、并发 batch 变深且 KV 读写压力增加。
- 方案：按任务分层路由短补全和长生成，限制低价值长输出；在灰度中对比 TPOT、吞吐、质量回归和成本，评估量化与投机解码收益。
- 结论：TPOT 是首字后体验的核心指标，不能被 TTFT 掩盖。

工程信号：指标、规模、系统边界、约束条件、验证结果。

## 常见误区
1. 把 TTFT 和 TPOT 混成一个“延迟”，导致优化方向错误。
2. 只优化首字时间，却忽略长输出场景的完整等待。
3. 看到流式输出就认为体验一定好，实际 token 间隔过宽仍会卡顿。
4. 用扩容解决所有慢问题，不拆分 Decode、batch 和 KV 读写瓶颈。

## 诊断题
**场景**：代码助手首字时间稳定在 1 秒左右，但开始输出后每几个字停顿一次。监控显示输出 token 数相近，TPOT 从 40ms 升到 120ms，KV 命中率没有明显下降。

**问题**：第一步应该优先看什么？

A. Decode 阶段的并发 batch、显存带宽和模型版本变化  
B. 是否应把系统提示词缩短一半以降低 Prefill  
C. 是否需要提高 temperature 让模型输出更自然  
D. 是否应该只看端到端总耗时，避免过度拆指标

**正确答案**：A

**解析**：A 直接对应 TPOT 上升和首字后卡顿。B 是强干扰项，缩短输入会改善 Prefill/TTFT，但题干显示首字稳定且 KV 命中没有下降。C 影响采样行为，不是性能第一排查。D 会掩盖 Prefill 与 Decode 的分工。

**排查路径**：
1. 分离 TTFT、TPOT、输出 token 数和端到端完成时间。
2. 按模型版本、并发、batch 和 GPU 实例拆分 TPOT。
3. 查看 Decode 阶段 KV 读写、显存带宽和队列等待。
4. 对长输出任务做路由和最大长度分层。
5. 灰度评估量化、投机解码或调度策略对质量和 TPOT 的影响。

## 核心结论
1. TTFT 回答“多久开始说”，TPOT 回答“说得快不快”。
2. TPOT 主要反映 Decode 阶段体验。
3. 长输出场景必须同时看 TPOT 和输出 token 数。
4. 优化 TPOT 要联动模型、batch、KV Cache、硬件和解码策略。
