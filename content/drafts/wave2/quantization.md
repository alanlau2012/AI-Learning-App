# quantization：量化

## 一句话定义
量化是用更低精度表示模型权重或激活，以降低显存和计算成本，并在质量可接受时提升部署密度和吞吐。

## 为什么重要
企业平台常常不是“有没有模型”，而是“同样预算能承载多少请求、延迟是否稳定、质量是否回归”。量化会影响显存占用、吞吐、TPOT、硬件兼容和模型质量。用户看到的可能是响应更快、并发更稳，也可能是边界任务准确率下降、格式稳定性变差或多语言表现退化。

## 心智模型
量化像把高清工程图压缩成更轻的版本。文件更小、传输和存储更省，但细节可能丢失。能不能用，不看压缩率本身，而看关键任务的质量细节有没有被压坏。

## 机制拆解
1. 模型原本可能用 FP16/BF16 等较高精度表示权重和计算。
2. 量化把部分权重或激活转换成 INT8、INT4 等低精度表示，降低显存占用。
3. 显存占用下降后，同一硬件可部署更大模型或更多并发实例。
4. 不同量化策略对速度、显存、精度和硬件支持影响不同，不能只看位宽。
5. 质量风险常出现在数学、代码、长上下文、多语言和严格格式任务。
6. 上线必须做任务级质量回归、吞吐/TPOT 对比和异常样本抽检。

## 企业案例
**标题：客服模型量化后成本下降但代码任务退化**

- 场景：某平台把 14B 模型从 FP16 切到 8-bit 量化，目标是在同样 GPU 集群上承载更多租户。
- 问题：客服问答吞吐提升 42%，显存占用下降约 35%；但代码解释任务的单元测试通过率从 86% 降到 78%，格式错误率上升。
- 分析：量化对通用客服语义影响较小，但对代码细节、符号和格式边界更敏感；平台最初只看平均满意度和成本，没有按任务回归。
- 方案：按任务建立质量回归集，对客服默认启用量化模型，对代码、财务计算和高风险合规任务保留高精度或更严格灰度。
- 结论：量化是部署和成本策略，不是无损加速开关。

## 常见误区
1. 认为位宽越低越好，只看显存节省，不看质量回归。
2. 一次量化全平台上线，不按任务风险分层。
3. 只测平均问答质量，不测代码、数学、格式和长上下文边界。
4. 忽略硬件和推理框架支持，导致理论压缩没有转化为实际吞吐。
5. 把量化收益和 batch、缓存、路由收益混在一起，无法判断真实来源。

## 诊断题
**场景**：平台把多个应用切到 4-bit 量化模型后，平均成本下降，但代码生成任务的测试通过率下降，客服 FAQ 基本不受影响。

**问题**：最合理的下一步是什么？

A. 继续全量推进量化，因为平均成本已经下降  
B. 只提高代码任务的 max output tokens，让模型补充更多解释  
C. 按任务风险做质量回归和路由分层，让敏感任务保留更高精度或灰度策略  
D. 把所有任务都改用更大 batch，提高吞吐抵消质量下降

**正确答案**：C

**解析**：C 同时处理量化收益和任务质量差异。A 是强干扰项，平均成本不能覆盖代码任务退化。B 增加输出不会恢复量化丢失的边界精度，还可能增加成本。D 优化吞吐，不解决质量回归。

**排查路径**：
1. 按任务类型比较量化前后的质量、格式错误、TPOT 和成本。
2. 找出对精度敏感的代码、数学、合规和长上下文样本。
3. 检查硬件和推理框架是否真正支持该量化策略。
4. 为低风险任务启用量化，为高风险任务保留高精度或加灰度。
5. 持续监控质量回归和成本收益。

## 核心结论
1. 量化用低精度换取显存、吞吐和部署密度。
2. 量化可能影响质量，尤其是边界和精细任务。
3. 企业平台要按任务做质量回归，而不是只看成本。
4. 量化收益必须结合硬件、框架和路由策略验证。
