# speculative-decoding：投机解码

## 一句话定义
投机解码是用较小或较快的草稿模型先生成多个候选 token，再由目标大模型并行验证，从而在命中率高时加速 Decode。

## 为什么重要
Decode 的逐 token 串行性会限制 TPOT。投机解码试图用“先草拟、再验证”的方式减少大模型逐步前进的次数。它主要影响 TPOT、吞吐和成本，但收益高度依赖草稿模型质量、接受率、验证成本、输出分布和系统实现。用户看到的是首字后输出更顺，或在命中低时反而更慢、更贵。

## 心智模型
投机解码像让助理先写几句草稿，专家一次性快速审阅：如果草稿大多对，专家就能更快往前；如果草稿经常错，专家不但要审，还要重写，反而浪费时间。

## 机制拆解
1. 草稿模型先根据当前上下文预测一串候选 token。
2. 目标大模型并行验证这些候选是否符合自己的分布。
3. 被接受的 token 可以一次性推进多个生成步，降低平均 TPOT。
4. 如果候选经常被拒绝，草稿计算和验证成本会抵消收益。
5. 接受率受任务类型、模型差距、采样策略、输出格式和领域分布影响。
6. 企业平台需要用线上回放评估质量、延迟、吞吐和成本，而不是只看理论加速比。

## 企业案例
**标题：客服摘要开启投机解码后收益不稳定**

- 场景：某 MaaS 平台为客服对话摘要服务开启投机解码，日均 80 万次长输出请求。
- 问题：通用摘要任务 TPOT 降低 32%，但金融投诉摘要几乎无收益，部分批次成本反而上升 9%。
- 分析：草稿模型在通用客服语料上接受率高，在金融投诉场景中术语、格式和合规措辞与目标模型差异大，候选 token 被频繁拒绝。
- 方案：按任务类型统计接受率、验证开销和质量回归；只在高接受率任务启用投机解码，低接受率场景改用模型路由或输出结构优化。
- 结论：投机解码不是让模型“猜答案”，而是系统级 Decode 加速策略，必须按任务分层启用。

工程信号：规模、指标、系统边界、约束条件、验证结果。

## 常见误区
1. 认为投机解码会改变答案质量，本质上目标模型仍负责验证和接受。
2. 只看平均加速，不看接受率低的任务和 P99 退化。
3. 草稿模型越小越好，忽略候选质量下降带来的拒绝成本。
4. 全量启用，不按任务类型、输出格式和领域分布分层。
5. 把投机解码当成 TTFT 优化，忽略它主要作用于 Decode/TPOT。

## 诊断题
**场景**：平台给所有长输出任务开启投机解码。通用客服摘要变快，但金融投诉摘要 TPOT 基本不变，成本上升。日志显示候选 token 接受率很低。

**问题**：最应该优先调整什么？

A. 把 max output tokens 提高，让投机解码有更多发挥空间  
B. 把温度提高，让草稿模型生成更多样候选  
C. 增加 Prefill 缓存，优先降低首字时延  
D. 按任务类型统计接受率和验证成本，只在高接受率场景启用或替换草稿模型

**正确答案**：D

**解析**：D 直接处理接受率低导致收益消失的问题。A 会增加输出长度和成本，不保证接受率提升。B 可能让候选更偏离目标模型。C 优化 Prefill/TTFT，而题干是 Decode 加速收益不稳定。

**排查路径**：
1. 按任务、模型版本和输出格式统计接受率。
2. 拆分草稿生成成本、目标验证成本和实际 TPOT。
3. 回放低接受率样本，判断是领域术语、格式还是采样策略导致。
4. 为高接受率任务灰度启用，低接受率任务关闭或换草稿模型。
5. 同步检查质量回归和成本变化。

## 核心结论
1. 投机解码用草稿模型提议 token，由目标模型验证。
2. 收益来自高接受率，不来自“猜得大胆”。
3. 它主要优化 Decode/TPOT，不是 Prefill 的替代方案。
4. 企业平台应按任务接受率、质量和成本分层启用。
