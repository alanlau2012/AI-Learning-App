# pd-separation：P-D 分离

## 一句话定义
P-D 分离是把 Prefill 和 Decode 两类资源特征不同的推理负载拆开调度，以提升资源利用率、吞吐和稳定性。

## 为什么重要
Prefill 更像大矩阵并行计算，Decode 更受逐 token 生成、KV 读写和串行性影响。把两者混在一组资源里，长上下文请求可能拖慢首字，长输出请求又可能占住 Decode。企业平台关心的是 TTFT、TPOT、GPU 利用率、KV 传输开销、队列长度和故障边界；P-D 分离能优化这些指标，但也会引入链路编排复杂度。

## 心智模型
把推理工厂拆成“读材料车间”和“逐字产出车间”。读材料适合大批量并行，逐字产出需要稳定接力。两个车间分开后，各自更高效；但中间要搬运半成品，调度和故障定位也更难。

## 机制拆解
1. Prefill 处理完整输入上下文，计算密集且可并行，常影响 TTFT。
2. Decode 按 token 逐步生成，更受 KV Cache 读写、显存带宽和输出长度影响。
3. P-D 分离把 Prefill 请求和 Decode 请求放到不同资源池或调度队列中。
4. 分离后可为长输入和长输出分别配置资源、batch 和优先级，提高整体稳定性。
5. 代价是需要在阶段间传递 KV 或中间状态，并处理失败重试、路由一致和观测追踪。
6. 如果链路编排不清，故障会从单点性能问题变成跨池排查问题。

## 企业案例
**标题：长文档问答拖慢短会话生成**

- 场景：某 MaaS 平台同时服务知识库问答和代码补全，每日约 260 万次请求，其中 15% 请求携带超长文档。
- 问题：长文档问答上线后，代码补全 TPOT 和 P99 明显波动；GPU 利用率看似很高，但短请求体验下降。
- 分析：长输入请求在同一资源池占用 Prefill，长输出请求继续占用 Decode，两个阶段互相挤压；平台只有端到端延迟，缺少 Prefill/Decode 分段队列指标。
- 方案：把长输入 Prefill 迁入独立资源池，对 Decode 池设置短交互任务优先级；增加 KV 传输监控和跨池 trace。
- 结论：P-D 分离能提升稳定性，但必须补齐阶段间观测和回退策略。

工程信号：规模、指标、系统边界、错误路径、约束条件、验证结果。

## 常见误区
1. 认为 P-D 分离只是把服务拆成两个进程，忽略 KV 和路由编排。
2. 只看 GPU 利用率，不看 Prefill 队列、Decode 队列和阶段间等待。
3. 为所有应用统一启用 P-D 分离，不评估请求长度分布和收益。
4. 忽略失败重试和跨池 trace，导致故障定位更困难。

## 诊断题
**场景**：长文档问答上线后，短代码补全的 TPOT 和 P99 都变差。监控只有端到端延迟，无法区分 Prefill 排队和 Decode 排队。

**问题**：最合理的第一步是什么？

A. 直接全量启用 P-D 分离，把所有请求拆到两个池  
B. 先补齐 Prefill/Decode 分段指标和请求长度分布，再评估分池与优先级策略  
C. 把长文档问答的 temperature 降低，减少输出随机性  
D. 只扩容当前统一推理池，避免架构复杂化

**正确答案**：B

**解析**：B 先建立分段观测，再决定是否 P-D 分离和如何分池。A 是强干扰项，方向可能对，但在没有分段指标和长度分布前全量拆分风险很高。C 不解决资源阶段挤压。D 可能缓解总容量，但无法处理长短请求互相干扰和阶段差异。

**排查路径**：
1. 为请求记录输入 token、输出 token、Prefill 队列、Decode 队列和阶段耗时。
2. 按任务类型分析长输入与长输出对 TTFT/TPOT 的影响。
3. 评估 Prefill 池和 Decode 池的资源配置与 KV 传输成本。
4. 为短交互任务设置 Decode 优先级。
5. 灰度验证 P99、吞吐、成本和故障恢复路径。

## 核心结论
1. Prefill 和 Decode 的资源特征不同，混跑会产生互相干扰。
2. P-D 分离能改善利用率和稳定性，但不是免费优化。
3. 阶段间 KV 传输、路由和 trace 是落地难点。
4. 上线前必须先有分段指标和灰度回归。
