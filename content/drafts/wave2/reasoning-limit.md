# reasoning-limit：推理能力边界

## 一句话定义
推理能力边界是模型在长链路任务、隐含约束、反事实判断、数学精确性和跨文件一致性上会失真的范围；它由上下文组织、任务拆解、工具可用性、反馈闭环和验证机制共同决定。

## 为什么重要
企业 AI 应用不能把模型当成一次性正确的决策者。推理边界一旦被忽略，用户看到的是“前面说得很像对，后面执行错了”：审批结论漏约束、Agent 改错文件、运营分析把口径混用、合规问答忽略例外条款。平台侧要看任务成功率、步骤级错误率、人工升级率、工具调用成功率、评测集回归和跨轮一致性，而不是只看单轮回答是否流畅。

## 心智模型
模型的推理像一个很强的临时分析师，但桌上资料、问题拆法、可用工具和复核机制决定了它能不能把复杂案子办完。短题里它可能表现很好；一旦任务跨越多份材料、多个约束和多轮动作，就必须把“聪明回答”改造成“可检查流程”。

## 机制拆解
1. 模型先在给定上下文内完成模式匹配和链式生成，但不会天然知道哪些约束必须跨步骤保持。
2. 上下文过长、证据冲突或关键条件被压缩掉时，模型会用看似合理的推断补空白。
3. 任务拆解不清会让模型在子目标之间漂移，例如先解释、再计划、再执行时口径逐步偏离。
4. 工具和外部校验能把部分推理转化为可验证操作，例如查询数据、运行测试、检查权限。
5. 反馈闭环决定错误是否被拦住：无评测、无 trace、无人审的链路会把早期误判继续放大。
6. MaaS 或 Agent 平台要把复杂任务拆成计划、执行、检查、升级，而不是只放大 prompt。

## 动画脚本或无动画说明
本讲为纯文本，不新增动画。推理边界横跨任务拆解、工具、评测和人审，不适合硬塞到现有单一画布；后续若做，应由 Agent Loop 或 Trace 类动画扩展，不在本轮修改协议。

## 企业案例
**标题：财务分析 Agent 在跨表口径中得出错误结论**

- 场景：某集团把月度经营分析交给 Agent，任务涉及 8 张报表、3 个区域口径和一份例外政策，每月约 400 次分析请求。
- 问题：模型生成的文字结构清晰，但把“确认收入”和“开票收入”混用，导致两个区域的毛利率解释错误；人工复核发现复杂任务首版通过率只有 71%。
- 分析：上下文里同时存在历史口径和新口径，任务没有拆成“口径确认 -> 数据查询 -> 计算校验 -> 文字生成”，模型在长链路里用语言连贯性补齐了缺失验证。
- 方案：把任务拆为四个检查点：先确认口径和期间，再用工具取数，再运行公式校验，最后生成结论；高金额差异必须触发人工复核。
- 结论：推理能力边界不是模型笨，而是复杂任务缺少可验证中间态。

工程信号：规模、指标、错误路径、约束条件、验证结果。

## 常见误区
1. 认为模型在短问答中表现好，就能稳定完成长链路企业任务。
2. 把错误全部归因于模型能力，不检查上下文、工具权限、任务拆解和验证点。
3. 用更长 prompt 堆约束，却不建立步骤级检查和失败升级。
4. 把“让模型解释推理过程”当成可靠验证，忽略外部数据和测试结果。
5. 只看最终答案正确率，不看中间步骤错误率和人工接管位置。

## 诊断题
**场景**：财务 Agent 在 8 张报表上生成经营分析，文字流畅，但抽检发现它混用了确认收入和开票收入。日志显示模型没有调用公式校验工具，直接生成结论。

**问题**：最应该优先修复什么？

A. 换成参数更大的模型，提升复杂推理能力  
B. 把所有财务制度全文加入系统提示词  
C. 要求模型在答案里展示更完整的思考过程  
D. 把任务拆成口径确认、取数、公式校验、结论生成，并对高风险差异加人审

**正确答案**：D

**解析**：D 把一次性生成变成可验证流程，直接覆盖口径混用和缺校验问题。A 是强干扰项，更大模型可能改善部分推理，但不能保证口径和公式被验证。B 会增加上下文负担，仍可能混用。C 展示过程不等于外部验证，甚至可能把错误解释得更像真的。

**排查路径**：
1. 找出错误发生在口径识别、取数、计算还是表达阶段。
2. 查看是否调用了数据查询、公式校验和版本化规则工具。
3. 为高风险报表建立步骤级 trace。
4. 把复杂任务拆成可回放检查点。
5. 用历史错例评测任务通过率和人工升级率。

## 核心结论
1. 推理边界由模型、上下文、任务拆解、工具和验证机制共同决定。
2. 企业应用不能假设模型一次性正确，必须设计检查点。
3. 长链路任务要把推理转化为可观测、可回放、可升级的流程。
4. 评测集和人工复核是推理边界的工程护栏。
