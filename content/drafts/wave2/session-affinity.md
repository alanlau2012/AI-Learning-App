# session-affinity：Session 亲和

## 一句话定义
Session 亲和是让同一会话的连续请求尽量路由到持有相关上下文和 KV Cache 的实例，以提升缓存命中和上下文连续性。

## 为什么重要
Session 亲和不是“粘住用户”这么简单，而是为了让多轮对话、Agent 任务和长上下文链路保持性能与状态连续。亲和失效时，用户看到的是首字忽快忽慢、上下文像丢了、限流策略不一致、同一任务重复计算。平台侧要看 KV Cache 命中率、会话迁移率、实例热点、P99 TTFT、故障切换成功率和亲和过期策略。

## 心智模型
会话像一份正在办理的案卷。每次都回到同一个柜台，柜台有之前读过的材料和处理记录；如果每轮都随机换柜台，就会反复翻案卷，甚至丢掉一些临时状态。但永远只找一个柜台也会造成排队和单点风险。

## 机制拆解
1. 首轮请求经过 Prefill 后，会在实例上形成 KV Cache 和会话相关状态。
2. 后续请求如果路由到同一实例或同一缓存域，可复用前缀缓存，降低重复 Prefill。
3. 如果请求被打散到不同实例，缓存未命中会导致 TTFT 上升，并可能触发重复上下文处理。
4. 亲和策略需要与负载均衡、扩缩容、限流、故障转移和租户隔离一起设计。
5. 过度亲和会造成热点实例、负载不均和容灾困难，因此要设置 TTL、迁移条件和降级路径。

## 动画脚本或无动画说明
本讲复用 `kv-cache`，不新增动画组件或协议。画面使用同会话命中 vs 打散重算的双路径：`session`、`instance` 表示会话与实例；`kv-write` 表示缓存写入；`cache-hit` 表示亲和命中；`route-miss`、`cache-miss` 表示打散后重算；`memory`、`eviction` 表示过度亲和带来的容量风险。

## 企业案例
**标题：多轮客服会话高峰期首字抖动**

- 场景：某 MaaS 平台承载 60 个业务应用，客服助手单日约 35 万轮对话，高峰期自动扩容。
- 问题：用户反馈同一会话前几轮很快，后几轮首字突然从 900ms 升到 4.5s；监控显示会话迁移率从 8% 升到 37%，KV 命中率下降到 22%。
- 分析：扩容后负载均衡只按实例空闲度分配，没有保持会话亲和；部分限流状态也跟着实例变化，导致体验不稳定。
- 方案：引入基于 session id 的一致性路由和缓存域，设置热点迁移阈值与亲和 TTL；扩容时预热热门会话并监控迁移率。
- 结论：Session 亲和提升缓存命中，但必须和热点、容灾、限流一起治理。

## 常见误区
1. 把 Session 亲和理解成用户粘性，而不是缓存和状态连续性。
2. 只追求亲和命中率，忽略热点实例和负载不均。
3. 扩缩容时不迁移或预热缓存，导致高峰期大量重算。
4. 会话状态、限流和缓存策略分别设计，出了问题难以复盘。

## 诊断题
**场景**：MaaS 平台扩容后，同一客服会话的 TTFT 波动明显。KV 命中率下降，会话迁移率上升，但 GPU 利用率并未持续打满。

**问题**：最应该优先排查哪项？

A. 是否需要把所有请求固定到最空闲的实例  
B. 扩容后的 session 路由是否打散了持有 KV Cache 的会话  
C. 是否应该提高输出 token 上限，让回答更完整  
D. 是否需要把 temperature 降到 0

**正确答案**：B

**解析**：B 直接对应会话迁移率上升和 KV 命中下降。A 是强干扰项，按空闲实例分配看似均衡，但会破坏会话缓存。C 影响输出长度和 TPOT，不解决首字抖动。D 影响随机性，不解释缓存命中下降。

**排查路径**：
1. 按 session id 追踪连续请求落到哪些实例。
2. 对比命中与未命中的 TTFT、Prefill 时长和缓存状态。
3. 检查扩缩容、故障转移和负载均衡策略是否绕过亲和。
4. 识别热点会话与实例负载不均。
5. 设置亲和 TTL、迁移条件、缓存预热和故障降级。

## 核心结论
1. Session 亲和的核心价值是 KV Cache 命中和上下文连续。
2. 请求打散会带来重复 Prefill、TTFT 抖动和状态不一致。
3. 亲和策略必须平衡缓存收益、热点风险和容灾能力。
4. 平台要把会话迁移率与缓存命中率作为一组指标看。
