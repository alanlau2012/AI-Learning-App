# batch-scheduling：Batch 调度

## 一句话定义
Batch 调度是在推理服务中把多个请求组织成批处理，并在吞吐、延迟、公平性和稳定性之间做权衡的系统策略。

## 为什么重要
Batch 决定 GPU 是否吃饱，也决定用户请求是否在队列里等太久。平台负责人看到的是吞吐、P99、排队时长、TPOT、超时率、租户公平性和成本；用户看到的是“有时很快、有时突然卡住”。Batch 调度出问题时，扩容不一定解决，因为瓶颈可能在队列策略、优先级、连续批处理或长短请求混排。

## 心智模型
推理服务像一组电梯。一次只载一个人响应最快但浪费运力；等满一电梯再走吞吐高但有人会等很久。好的调度不是永远大 batch 或小 batch，而是按任务、SLA 和队列状态动态决定谁一起走、谁优先走、谁不能拖垮别人。

## 机制拆解
1. 请求到达后先进入队列，调度器按时间窗、优先级和资源状态组成 batch。
2. 大 batch 能提高吞吐和 GPU 利用率，但会增加单个请求的等待和尾延迟风险。
3. 小 batch 响应更快，但吞吐低、成本高，GPU 可能空转。
4. 连续批处理会在 Decode 过程中动态加入新请求，改善吞吐但需要复杂的 KV 和队列管理。
5. 长输出、长输入和高优任务混排时，会互相影响 TTFT、TPOT、P99 和公平性。
6. 企业平台需要按租户、任务类型和 SLA 做隔离、优先级和超时策略。

## 企业案例
**标题：知识库问答高峰期 P99 突然拉长**

- 场景：某企业 MaaS 平台服务 40 个内部应用，工作日上午知识库问答峰值达到每分钟 1.8 万请求。
- 问题：平均延迟只上升 18%，但 P99 从 6 秒升到 24 秒；投诉集中在客服和法务两个高优应用。
- 分析：平台为了提高吞吐把 batch 等待窗口调大，低优批量摘要和高优问答混在同一队列；长输出任务占用 Decode，导致短请求排队。
- 方案：拆分在线问答和离线摘要队列，为高优应用设置最大等待时间和保底并发；启用动态 batch 上限，按 P99 和吞吐共同调参。
- 结论：Batch 调度不是单纯提高吞吐，而是 SLA、成本和公平性的权衡。

## 常见误区
1. 认为 batch 越大越好，只看吞吐不看 P99 和用户等待。
2. 看到 P99 高就盲目扩容，不检查队列等待和任务混排。
3. 所有租户共用一个队列，忽略高优应用和离线任务隔离。
4. 只看平均延迟，掩盖尾部用户长时间等待。
5. 调度策略上线后不按任务类型回放评测，导致局部收益、整体退化。

## 诊断题
**场景**：平台把 batch 等待窗口调大后，总吞吐提升 28%，但客服应用 P99 从 5 秒升到 19 秒。日志显示客服问答与低优批量摘要共享队列。

**问题**：第一步最合理的处理是什么？

A. 继续增大 batch，让 GPU 利用率进一步提高  
B. 立即把所有请求都切到小 batch，牺牲吞吐换体验  
C. 按任务和 SLA 拆队列，限制高优请求最大等待，并观察 P99 与吞吐  
D. 只增加前端 loading 提示，降低用户焦虑

**正确答案**：C

**解析**：C 同时处理任务混排、优先级和尾延迟。A 是强干扰项，吞吐更高可能继续恶化高优 P99。B 方向过猛，会让离线和低优任务成本失控，也不是第一步。D 只改善感知，不解决调度瓶颈。

**排查路径**：
1. 拆分排队时间、Prefill、Decode 和输出 token 数。
2. 按租户、任务类型和优先级统计 batch 组成。
3. 查看长输入/长输出任务是否阻塞短请求。
4. 设置队列隔离、最大等待时间和动态 batch 上限。
5. 用 P50/P95/P99、吞吐和成本共同评估调度策略。

## 核心结论
1. Batch 调度是在吞吐、延迟和公平性之间做工程权衡。
2. 大 batch 不等于好体验，小 batch 不等于好成本。
3. P99、排队时间和任务混排是调度排查重点。
4. 企业 MaaS 要按租户、任务类型和 SLA 做调度隔离。
