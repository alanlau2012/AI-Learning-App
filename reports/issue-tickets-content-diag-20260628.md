## 5. 高风险逐项（P0/P1）

### [P1·系统性] 全局：约 22% 单选题存在"最长选项即正确答案"结构泄漏

- **原文片段**：见 §2.2 表格（12 道题，覆盖 value-review-agent / permission-governance / ops-diagnosis-agent / eval / requirement-decomposition-agent / test-generation-agent / reasoning-limit / code-review-agent / ai-native-org / multi-agent / context-window / repo-context）
- **问题类型**：正确答案系统性偏长 + 结构化措辞独特（"先 X、再 Y、并 Z"句式仅在正确答案出现）
- **为什么有问题**：诊断题的目的是检验用户能否从工程推理得出答案；当正确答案明显比干扰项长且更结构化时，用户可通过模式识别（"最长最全面 = 正确"）绕过推理，**评估有效性被削弱**。在企业培训场景下，受训者会迅速学会这一规律，导致后续 40+ 道题评估失真。
- **可能误导的工程判断**：受训者未来在真实事故复盘时可能误以为"看起来最全面、最结构化的方案就是对的"，而真实工程中"最小可执行下一步"往往比"全面方案"更重要（例如 ttft 题正确答案就是简短的 D）。
- **建议修复方向**：
  1. **优先**：把过长正确答案拆为核心动作 + 1 个限定条件（如 `value-review-agent` 的 C 可改为"把指标从生成量改为采纳率与单位价值成本"）。
  2. **次优**：将 1 个明显错误的干扰项升级为可信工程动作（参考 `model-gateway` 题中 SDK 埋点这种"看似合理但有缺陷"的强干扰项设计）。
  3. **过程性**：在写作模板中加入"4 个选项字数差 ≤ 4 字"的硬约束。
- **是否需要人工确认**：是。修复涉及内容写作，需主开发按 §5.1 文件所有权规则合入。

### [P1] permission-governance - 干扰项荒谬 + 正确答案最长（双重泄漏）

- **原文片段**：
  - 正确 A："收敛到最小权限默认只读，高风险写或删动作走人工审批并接入完整审计日志"（33 字，含 4 个动作）
  - 干扰 B："提高模型的安全对齐能力以抵抗注入"（14 字）
  - 干扰 C："增加 Agent 可访问的数据表以提升分析能力"（16 字，方向相反）
  - 干扰 D："给 Agent 增加一次自我确认的提示步骤"（15 字）
- **问题类型**：结构泄漏 + 干扰项质量差
- **为什么有问题**：B/C/D 都是单一动作且明显错误或方向相反；A 是唯一合理动作且字数是其他的 2 倍以上。用户即使完全不懂权限治理，也能通过"哪个最长最全面"猜对。
- **可能误导的工程判断**：让受训者误以为"安全对齐 / 自我确认"这类轻量动作在实际中无效——但实际工程中，安全对齐 + 最小权限 + 审批是组合拳，单独任一项都不是错的。本题把"组合动作"设为正确，单一动作设为错误，制造了虚假二分。
- **建议修复方向**：
  - 把 B 改为可信强干扰项，如"先在系统提示中加入防注入规则并收紧工具描述"——这是真实工程中常见的过渡措施，但不如权限收敛彻底。
  - 把 A 拆短，如"先收敛到最小权限默认只读，高风险写动作走人工审批"。
- **是否需要人工确认**：是。

### [P1] ttft - 多选题答案过于直接映射题干

- **原文片段**：
  - 题干："TTFT 从 800ms 上升到 4s...队列长度、RAG 检索耗时和 KV Cache 未命中率都上升。"
  - A "检查网关和推理服务的排队等待" → 队列长度
  - B "检查 RAG 检索和重排链路是否变慢" → RAG 检索耗时
  - C "检查会话亲和和 KV Cache 命中是否下降" → KV Cache 未命中率
  - D "只优化 Decode 阶段 TPOT..."（明显错误）
- **问题类型**：复杂工程问题被简化为"题干数据→选项映射"
- **为什么有问题**：题干已经把三个数据点逐字列出，每个数据点对应一个正确选项。多选题退化成了"找出与题干数据匹配的三个选项"，缺少需要推理的环节。受训者无法从此题学到"TTFT 上升时如何区分 Prefill 与 Decode 阶段"，因为题干直接给定了答案。
- **可能误导的工程判断**：受训者未来在真实场景中遇到"TTFT 上升但只有部分指标异常"时，可能仍机械地勾选所有看似相关的排查方向，而不会判断"哪些指标其实是噪声"。
- **建议修复方向**：
  - 选项中至少有 1 个"看似对应题干数据但应被排除"的迷惑项，例如："查看 GPU 显存利用率是否上升"——看起来合理但显存利用率与 TTFT 没有强因果关系（除非已经 OOM）。
  - 或将题干改为只给 2 个数据点，让用户判断"第三个数据点是否需要采集"。
- **是否需要人工确认**：是。

### [P1] maas / cost-routing / sla - 干扰项明显荒谬

- **共同问题**：三道题的 3 个干扰项均为"先给所有应用换最大模型 / 立刻全量切低成本模型 / 关闭 trace / 只把可用性提到 99.9%"这类明显错误或方向相反的选项。
- **为什么有问题**：题目退化为"找出唯一不荒谬的选项"，难度被弱化。
- **建议修复方向**：参考 `model-gateway` 题（B 选项"先要求 SDK 统一埋点并冻结新增直连"是强干扰项）、`attention` 题、`prefill` 题的强干扰项设计，把至少 1 个荒谬干扰项升级为"看似合理但有工程缺陷"的可信选项。

## 6. 修复优先级建议

按 ROI 排序：

1. **第一批（最高优先级，1-2 天）**：修复 §2.2 表格中 12 道结构泄漏题。每题 5 分钟，主要工作是"缩短正确答案核心动作"或"把 1 个荒谬干扰项升级为强干扰项"。这一批修复直接消除"最长即正确"的系统性问题。
2. **第二批（中优先级，0.5 天）**：修复 `maas` / `cost-routing` / `sla` / `permission-governance` 4 道题的荒谬干扰项。可复用 model-gateway 题的强干扰项设计模式。
