# 12 讲样板课程内容评审报告

## 1. 总体结论

**有条件通过：修完 P0 / P1 后可作为样板。**

一句话判断：这 12 讲已经具备“AI 应用工程教材”的基本口径和工程场景密度，但仍有会阻塞入库的结构问题，以及会被后续 44 讲复制的题目、案例、动画脚本模板风险。

## 2. 总体评分

- Schema 合规性：8.0 / 10
- 概念准确性：8.2 / 10
- 工程落地性：8.0 / 10
- 企业案例质量：7.4 / 10
- 诊断题质量：7.0 / 10
- 动画脚本质量：7.1 / 10
- 样板一致性：7.6 / 10
- 综合评分：7.6 / 10

## 3. P0 必须修改项

### P0-01：`attention` 存在悬空关联 id

- 涉及课程：注意力机制
- 涉及文件：`content/drafts/attention.json`
- 问题描述：`relatedConceptIds` 与 `diagnosticQuestion.relatedConceptIds` 使用了 `rag-pipeline`，但 `docs/content-schema.md` 的 56 讲登记表中不存在该 KnowledgePoint id。`rag-pipeline` 只是动画类型枚举之一，不是课程 id。
- 为什么严重：入库到 `src/data/*` 后会违反 `validate:structure` 的“关联无悬空”规则，阻塞内容入库与发布。
- 建议修改：将 `rag-pipeline` 移出 `relatedConceptIds`；若只是术语，放入 `tags` 或后续 glossary。关联课程可改为已存在的 `context-window`、`context-pollution`、`eval` 等。
- 是否需要人工确认：否。

### P0-02：`skill` 动画配置违反首版登记表与 registry 覆盖范围

- 涉及课程：Skill
- 涉及文件：`content/drafts/skill.json`
- 问题描述：`skill.json` 设置 `hasAnimation: true`，动画类型为 `skill-lifecycle`。该类型属于 `AnimationType` 扩展枚举，但不在首版 8 个 registry 组件中；并且 `docs/content-schema.md` 登记表将 `skill` 标为首版无动画。
- 为什么严重：若按当前状态入库，P3 后 `validate:animation` 会失败；更重要的是内容草稿越权改变了“首版哪些课含动画”的权威登记。
- 建议修改：首版入库时将 `skill` 改为 `hasAnimation:false` 并移除 `animation`；或由主开发正式修改登记表、实现 `SkillLifecycleAnimation` 并更新 registry。后者不建议作为本轮内容修订的默认路径。
- 是否需要人工确认：是。需要产品/主开发确认首版是否真的要新增 Skill 动画。

## 4. P1 建议修改项

### P1-01：诊断题干扰项普遍偏弱，部分题更像“送分题”

- 涉及课程：Token、Prefill、Decode、KV Cache、模型网关、多模型路由、上下文窗口、Agent Loop、Skill、Issue Fix Agent
- 涉及文件：`content/drafts/*.json`
- 问题描述：很多错误选项明显荒谬，例如“前端打字机动画”“把温度调高”“关闭所有测试”“写得更热情”。这会降低训练工程判断的价值。
- 为什么严重：诊断题是产品闭环核心。如果样板题目太容易，后续 44 讲会复制“真实场景 + 明显正确答案”的弱模式。
- 建议修改：每题至少保留 2 个工程上“看似合理但优先级错误”的干扰项，例如扩容、换模型、调整 RAG 数量、加缓存、改路由、加评测等，并在解析中说明优先级。
- 是否需要人工确认：否。

### P1-02：企业案例同质化偏高，过多集中在 RAG、TTFT、KV Cache、泛 MaaS 高峰期

- 涉及课程：Token、Attention、Prefill、TTFT、KV Cache、Context Window
- 涉及文件：`content/drafts/token.json`、`attention.json`、`prefill.json`、`ttft.json`、`kv-cache.json`、`context-window.json`
- 问题描述：案例质量总体合格，但上半部分课程大量使用“RAG 注入过多、首字变慢、上下文污染、缓存命中下降”的相近结构。
- 为什么严重：作为 56 讲样板，会诱导后续内容都落在同一类性能/检索问题上，削弱课程地图的层次。
- 建议修改：保留关键案例，但增加不同企业场景：客服 SLA、办公审批、代码仓库、权限审计、模型供应商切换、Agent 任务失败、合规审计等。
- 是否需要人工确认：否。

### P1-03：动画步骤描述混入“画面展示”口吻，仍需转换为可消费机制状态

- 涉及课程：全部含动画课程
- 涉及文件：`content/drafts/*.json`
- 问题描述：`AnimationStep.description` 中频繁出现“画面展示”“并排展示”等导演口吻。schema 允许 description，但 AnimationPlayer 需要的是当前步骤解释，画面意图应尽量由 `highlightTargets` 与动画组件映射承载。
- 为什么严重：后续如果照此写，会把内容字段变成视觉脚本，增加动画组件适配成本。
- 建议修改：保留 `highlightTargets`，把 description 改成“该步骤机制解释”。例如从“画面展示请求标签”改为“路由策略读取任务类型、敏感级别、时延预算和质量要求”。
- 是否需要人工确认：否。

### P1-04：部分概念表述需要增加边界条件，避免过度确定

- 涉及课程：KV Cache、注意力机制、多模型路由、Skill
- 涉及文件：`content/drafts/kv-cache.json`、`attention.json`、`multi-model-routing.json`、`skill.json`
- 问题描述：KV Cache 的多轮复用依赖服务端保留缓存、前缀缓存或会话亲和，并非所有 API 调用天然成立；注意力与“引用错误片段”的因果关系不能说得像可直接观测的解释；多模型路由中的“低成本优先”需要明确是在质量门槛满足后。
- 为什么严重：课程面向资深工程师和平台负责人，概念边界不清会被认为是简化过度。
- 建议修改：在相关机制或误区中补一句边界条件，区分“机制可能导致”与“日志可直接证明”。
- 是否需要人工确认：部分需要。尤其是 KV Cache 多轮复用口径需与主开发/平台教材口径一致。

### P1-05：`contentStatus: "mvp"` 对全部 12 讲过早

- 涉及课程：全部 12 讲
- 涉及文件：`content/drafts/*.json`
- 问题描述：所有草稿均标记为 `mvp`。从字段完整度看可作为候选，但内容质量尚未修完 P0/P1。
- 为什么严重：如果主开发按状态直接纳入完整内容，会绕过“有条件通过”的审核结论。
- 建议修改：修订前保持草稿状态，不进入 `src/data/*`；入库时仅将修完 P0/P1 的课程标为 `mvp`，其余降为 `demo` 或暂不入库。
- 是否需要人工确认：是。

## 5. P2 可优化项

- 统一每讲机制步数节奏：多数为 5-6 步，合适；后续模板可规定“定义 1 句、机制 5-6 步、误区 4-5 条、结论 4-5 条”。
- 给企业案例增加“可观测指标”字段口径，虽然 schema 没有单独字段，但可在 `analysis` 或 `solution` 中显式写出平台侧指标。
- 少用“闭环、治理、能力资产、质量回流”等高频词，避免 AI 味。
- 多选题目前只有 TTFT 一题，后续可以适度增加多选，但必须保证正确项逻辑清楚。
- `tags` 整体有用，但英文/中文混用需统一策略：建议每讲至少包含英文核心术语 + 所属系统域。

## 6. 逐讲评审

### 01｜Token

#### 结论

通过。

#### 主要优点

定义准确，明确说明 Token 不是字也不一定是词；能区分输入 Token 与输出 Token，并关联成本、上下文、Prefill、Decode、TTFT。

#### 主要问题

诊断题正确答案过明显，错误项缺少真实工程迷惑性。

#### 事实与概念风险

无明显事实错误。更换模型后分词器和计费口径需重新校准，这一点写得好。

#### 工程案例评价

RAG 成本失控案例可信，包含请求量、输入/输出 Token、检索片段与历史会话排查方向。

#### 诊断题评价

场景真实，但选项 `a/c/d` 太弱。建议加入“切换便宜模型”“限制 max output tokens”“增加缓存”等更真实干扰项。

#### 动画脚本评价

`token-flow` 类型正确，步骤清晰。description 中有少量“画面展示”口吻，但可实现。

#### 建议修改方向

保留主体，强化诊断题干扰项即可作为模板。

### 02｜注意力机制

#### 结论

有条件通过。

#### 主要优点

明确避免把注意力说成人类意识；讲清 Query / Key / Value、权重、上下文污染和长上下文干扰。

#### 主要问题

`rag-pipeline` 悬空引用是 P0；部分“注意力偏移导致引用旧文档”的表述需要强调这是机制解释，不是普通日志能直接证明的事实。

#### 事实与概念风险

概念方向正确，但要避免把注意力权重等同于可直接解释模型行为的完整因果证据。

#### 工程案例评价

合同问答引用旧制度的案例好，场景明确，能落到版本过滤、权威来源标记和评测集。

#### 诊断题评价

题目方向好，但答案较明显。可增加“调低温度 + 引用来源校验”“rerank 加权”之类更强干扰项。

#### 动画脚本评价

`attention-map` 类型正确；热力图、连线、高亮均可实现。注意不要做复杂 dashboard。

#### 建议修改方向

先修关联 id；再将“注意力被错误材料牵引”改为更稳妥的工程口径。

### 03｜Prefill

#### 结论

通过。

题干数据充分，正确答案清楚；干扰项偏弱。

#### 动画脚本评价

`prefill-decode` 复用合理，步骤能被播放器消费。

#### 建议修改方向

#### 动画脚本评价

`skill-lifecycle` 是合法扩展枚举，但不是首版 registry；当前不应作为 MVP 1.0 入库动画。

#### 建议修改方向

### 12｜Issue Fix Agent

#### 结论

通过，适合作为样板。

#### 主要优点

完整覆盖问题单理解、规范化、定位、修改、测试、PR、人审和反馈回流，没有写成“AI 自动改 bug”。

#### 事实与概念风险

无明显事实错误。

#### 工程案例评价

## 8. 内容重复与风格问题

12 讲风格基本一致，结构完整，语言偏工程教材，不像营销课或提示词技巧合集。主要重复问题是：

- RAG 片段过长、TTFT 变慢、上下文污染、KV Cache 命中下降在前 6 讲反复出现。
- 诊断题套路重复：题干给出明确指标，正确答案对应唯一概念，错误项明显不专业。
- 常见误区和结论中反复出现“闭环、治理、观测、评测、回流”，有轻微 AI 味。
- 动画 description 大量以“画面展示”开头，样板化后会污染内容字段。

## 9. 适合作为样板的课程

1. TTFT：端到端链路拆分最完整，诊断题也最好。
2. 模型网关：平台治理口径清楚，不是 API 转发科普。
3. Agent Loop：很好体现 Agent 是带状态、工具、校验和退出条件的系统。
4. Issue Fix Agent：最贴合“AI 原生软件工程闭环”，案例和机制完整。
5. Token：基础概念讲出了成本、上下文和性能治理，不停留在分词科普。

## 10. 不适合作为样板的课程

- Skill：内容方向可用，但因动画/登记冲突，不适合在修复前作为样板。
- 注意力机制：概念讲解合格，但存在悬空关联；且“注意力导致引用错误”的表述需要更谨慎。
- Prefill / KV Cache：质量较高，但案例模式与 TTFT 高度相近，若作为模板需补充差异化案例。

## 11. 修订优先级

1. 先修 schema 问题：`attention` 悬空 id、`skill` 动画登记冲突。
2. 再修概念边界：KV Cache 多轮复用、注意力解释边界、多模型路由质量门槛。
3. 再修诊断题：把明显错误选项改为真实工程干扰项。
4. 再修动画脚本：description 回归机制解释，画面意图交给 `highlightTargets`。
5. 再修企业案例：减少 RAG/TTFT/KV 的重复场景。
6. 最后统一语言风格：减少“闭环/治理/回流”的套话密度。

## 12. 给内容生产 Agent 的修改建议

- 每讲诊断题至少设置 2 个“工程上可能被误选”的干扰项，不要使用明显荒谬选项。
- `relatedConceptIds` 只能写 56 讲登记表里的 id；动画类型、术语、系统名不要混入。
- 动画步骤只写当前机制状态和变化，不写复杂视觉导演说明。
- 企业案例必须包含场景、现象、平台指标、分析和解决方向；不要只写抽象治理。
- 写 KV Cache、注意力、路由等机制时，必须补充适用条件和边界。
- 不要为了显得高级反复使用“闭环、治理、资产、回流”；优先写可观测指标和排查动作。

## 13. 给主开发 Agent 的接入建议

- 可以先接入：`token.json`、`ttft.json`、`model-gateway.json`、`agent-loop.json`、`issue-fix-agent.json`，但建议先修诊断题干扰项。
- 必须修完再接入：`attention.json`（悬空 id）、`skill.json`（动画/登记冲突）。
- 可接入但建议同步修订：`prefill.json`、`decode.json`、`kv-cache.json`、`multi-model-routing.json`、`context-window.json`。
- 是否需要临时 fallback：如果 `skill` 保留 `skill-lifecycle`，AnimationPlayer 会需要未注册类型 fallback；但不建议以 fallback 方式发布 MVP 内容。
- 对 ConceptPage 的影响：字段结构总体符合页面渲染；长文本长度适中。
- 对 AnimationPlayer 的影响：除 `skill-lifecycle` 外，动画类型均在首版 registry；description 目前可展示，但应修成机制解释。
- 对 DiagnosticQuestion 的影响：结构可渲染，判题逻辑合法；质量层面需增强干扰项。

- 是否可以让主开发接入部分内容：可以接入修完 P0 后的 5-8 讲作为 MVP Demo / 1.0 候选，但不要整批直接入库。
