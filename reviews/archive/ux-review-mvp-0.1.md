# MVP 0.1 产品体验复盘报告

> 复盘角色：产品负责人 / 学习体验设计师 / 课程主编（只读评审，不改代码、不改内容）。
> 评审对象：已封版的 12 讲 MVP 0.1。
> 评审依据：`design.md`（视觉与体验最高优先级）、`docs/*`、`reports/mvp-0.1-summary.md`、`reports/e2e-verification-12-lessons.md`，以及 `src/*` 实际实现与 `src/data/*` 实际内容。
> 评审口径：端到端验证只证明"能跑"，本轮判断"值不值得作为后续 56 讲样板"。

---

## 1. 总体结论

**结论：有条件通过 —— 修复关键体验问题后可作为 12 讲产品样板。**

一句话原因：产品的**气质方向是对的**（内容是真正的工程教材、视觉严格符合 design.md、诊断题有工程判断力），但有三类问题会在扩展 44 讲时被批量复制或在演示时直接暴露：①课程内导航频繁把用户送进空白 stub 页（12 讲闭环在自然点击下是"漏的"）；②11 个动画里只有 KV Cache 是真正的机制可视化，其余 7 类全部回退到同一个"通用动画"，且把内部英文 key 直接铺给用户；③`contentStatus`、动画 `type`、`highlightTargets` 等内部标识泄漏到 UI。这些都不是"产品方向跑偏"，而是"样板定稿前必须收口"的问题。

---

## 2. 产品定位判断

- **当前产品最像什么**：一本结构稳定、正文克制、带工程案例与诊断题的**可交互 AI 应用工程教材**。首页是入口而非报表，详情页是长阅读而非卡片墙，左侧目录稳定，动画是章节内的一个小节。方向正确。
- **是否符合"轻量工程学习书桌"**：是。第一屏只有"继续学习"一个核心动作 + 简洁进度 + 4 条推荐路径 + 模块入口列表，没有 dashboard 化。
- **是否偏向 dashboard / 百科 / 课程页 / 博客**：都不明显。
  - 不是 dashboard：首页无大量统计图表，统计被正确地放在 Profile。
  - 不是百科：每讲有"为什么重要 / 企业案例 / 误区 / 诊断题"，是判断力训练而非词条。
  - 不是营销页：没有 hero 巨字、没有 CTA 轰炸、没有渐变光效。
  - 唯一轻微偏移：详情页头部把 `mvp` 这种内部成熟度标签暴露给用户，略有"半成品后台"气味（见 P1-03）。
- **最大的产品气质问题**：不是"长什么样"，而是"点下去会掉进空房间"。12 讲分散在 6 个模块里，模块页里 80% 是 stub 卡片、详情页"下一个"按钮 9/12 会跳进 stub。教材的"页与页之间"是断的，这是当前最伤气质的体验问题。

---

## 3. 首页体验评审

- **优点**：
  - 第一屏信息密度低，主动作唯一（继续学习 · Token），完全符合 design.md §6。
  - 进度面板克制（`0 / 56` + 完成度 + 收藏 + 连续天数），没有堆统计。
  - 推荐路径只露 4 条、模块只做入口行（标题 + done/total），没有强卡片堆叠。
  - 一眼能看懂"这是一个把 AI 应用工程从概念带到落地的学习系统"。
- **问题**：
  - "继续学习"对新用户指向 Token（正确）；但一旦用户访问过任意 stub（例如从模块页点进 `Transformer`），`lastVisitedConceptId` 会让"继续学习"回到一个空白页。即首页核心动作可能把用户带回空房间（根因同 P0-01）。
  - "六大模块"列表的 done/total 对 12 讲样板会长期显示 `0/10`、`1/10` 这类低分母完成度，演示时观感偏"未完成"，但这是 stub 现状导致，可接受。
- **是否阻塞样板**：否（首页本身达标，问题来自其指向的 stub）。
- **建议**：首页保持现状；配合 P0-01 让"继续学习/推荐"在 MVP 阶段优先指向有正文的 12 讲。

## 4. 知识点详情页体验评审

总体：12 讲正文质量明显高于"AI 生成的文章堆叠"，是站在工程场景写的，定义—为什么重要—心智模型—机制—案例—误区—诊断题—结论—关联的节奏适合长阅读，正文宽度受控（860px）。下面是 4 讲精评 + 8 讲抽查。

### Token

- 阅读节奏：好。从"不是字也不是词"切入，迅速落到成本/上下文/TTFT/缓存，长度适中。
- 工程判断力：强。把 Token 定位为"业务体验翻译成工程指标的入口"，区分输入/输出 Token 对 Prefill 与 Decode 的不同影响，是负责人真正需要的视角。
- 内容层次：清楚。机制 6 步、误区 5 条、结论 5 条，层层递进。
- 页面呈现：达标。一句话定义有蓝色竖线，机制有 mono 序号。
- 主要问题：动画区只是"通用动画"（见第 5 节），把 `input-text / tokenizer / ttft / cost` 等英文 key 直接当标签铺出来，和正文中文气质割裂；"下一个"跳到 `semantic-space` 空白页。
- 建议：动画专项；导航跳过 stub。

### KV Cache

- 阅读节奏：好，是全场最完整的一讲。定义里点出"是否能跨请求复用取决于具体服务实现"，避免把 KV Cache 讲死，专业度高。
- 工程判断力：强。命中率、Session 亲和、显存挤占、扩缩容冷启动之间的因果链交代清楚。
- 内容层次：清楚，机制 6 步、动画 5 步、案例完整、误区 5 条。
- 页面呈现：最佳。**唯一**拥有专用机制画布（Session→Prefill→KV Cache→命中/未命中），动画真正解释了机制。
- 主要问题：`src/data/concepts.ts` 内联还保留了一份旧的 `kv-cache`（`contentStatus: 'demo'`），被 `demoConcepts.ts` 的 `mvp` 版覆盖成死数据（见 P1-04）；"下一个"跳到 `session-affinity` 空白页。
- 建议：清理重复数据；以本讲为内容与动画的黄金样板。

### Agent Loop

- 阅读节奏：好。Observe→Plan→Act→Check→Continue/Stop 的循环讲得清楚，并特意说明"有些系统叫 Think，但工程上更关注可执行可验证"，口径成熟。
- 工程判断力：强。把 Agent 的关键从"能调用工具"重新定义为"能在循环里被约束、验证和停止"，案例（运维 Agent 重复重启）非常贴合企业现实。
- 内容层次：清楚，6 步动画含人审分支。
- 页面呈现：达标，但动画是通用动画——`goal / observe / plan / act / check / human-review` 这种本应成为环形画布节点的 key，被退化成一排英文标签，循环感完全没有可视化出来。这是最"该有专用动画却没有"的一讲。
- 主要问题：动画解释力与内容深度严重不匹配；"下一个"跳到 `tool-calling` 空白页。
- 建议：列为动画专项第一优先（环形 Loop 画布）。

### Issue Fix Agent

- 阅读节奏：好。理解→定位→最小修改→验证→PR→人审→质量回流，链路完整。
- 工程判断力：强。点破"上限不只取决于模型能力，也取决于问题单和工程流程质量"，案例（问题单只有一句"页面偶现报错"）极具代入感。
- 内容层次：清楚，6 步动画含质量回流闭环。
- 页面呈现：达标，但同为通用动画，6 步闭环（issue→repo-context→patch→tests→pr→feedback）退化成英文 key 列表。
- 主要问题：动画未体现"失败回到定位而非盲目扩散"的关键分支；"下一个"跳到 `requirement-decomposition-agent` 空白页。
- 建议：动画专项（带回退箭头的流程图）。

### 其余 8 讲抽查结论

- **注意力机制**：质量高，难得地强调"注意力权重不能当完整因果解释""答案流畅≠依据正确"，RAG 引用过期条款的案例很真实。动画为通用动画，`pollution / shifted-weights / rerank` 这类本可做高亮对比的 key 被铺成标签。建议动画专项。
- **Prefill / Decode / TTFT**：三讲互相呼应，TTFT 做了端到端链路拆解、Decode 区分 TPOT 与总时延、Prefill 强调"减少无效上下文比扩容更有效"，工程判断力一致在线。三者共用 `prefill-decode` 通用动画，时间轴/首字前后拆分没有被可视化，解释力偏弱。
- **模型网关 / 多模型路由**：站位正确（控制面、按约束派单、评测与观测回流），不是 API 转发科普。共用 `model-router` 通用动画，路由"过滤门槛→比较成本/时延"的核心判断没画出来。
- **上下文窗口**：把"窗口≠长期记忆""硬约束应固定在高优先级上下文"讲透，案例（代码 Agent 改了公共接口）好。动画通用。
- **Skill**：唯一明确无动画（`hasAnimation:false`），页面友好降级为"暂无动画配置"，正文质量不受影响，作为对照反而说明：**降级文案比 7 个泄漏 key 的通用动画更干净**。

**共性结论**：12 讲内容本身可以直接作为 56 讲的写作样板——结构稳定、口径成熟、案例可信、不堆砌。唯一的内容层风险是**模板化痕迹**（机制几乎都是 6 条、结论都是 5 条、心智模型都以"可以把 X 理解为…"开头），长期连读会略显工整单调，属 P2。

## 5. 动画体验评审

事实基础：动画 registry 中只有 `kv-cache` 指向专用组件 `KVCacheAnimation`，其余 7 类（`token-flow / attention-map / context-window / prefill-decode / model-router / agent-loop / issue-fix-flow`）**全部指向同一个 `GenericMechanismAnimation`**。通用动画渲染 = 一行 `config.type` 文本 + 一排进度圆点 + 当前步标题/描述 + **把 `highlightTargets` 内部英文 key 原样当标签铺出来**。而同样的步标题/描述在播放器下方 caption 又出现了一次（重复展示）。

| 动画类型 / 课程 | 当前状态 | 体验判断 | 建议 |
|---|---|---|---|
| `kv-cache`（KV Cache） | 专用画布，节点+命中/未命中 | 真正解释机制，是样板 | **可以保留**（小修：减弱 reduced-motion 下的视觉、对齐配色） |
| `token-flow`（Token） | 通用动画 | 未可视化"文本→分词→编号→Prefill/Decode→成本"流，仅铺 key | **需要专项重做**（横向流水线） |
| `attention-map`（注意力机制） | 通用动画 | 未可视化权重/污染/重排对比 | **需要专项重做**（权重热度/连线） |
| `prefill-decode`（Prefill/Decode/TTFT） | 通用动画（3 讲共用） | 未可视化首字前/后时间轴拆分 | **需要专项重做**（时间轴 + 阶段拆分，最高复用价值） |
| `model-router`（模型网关/多模型路由） | 通用动画（2 讲共用） | 未可视化"过滤门槛→择优→降级" | **需要专项重做**（路由分支图） |
| `agent-loop`（Agent Loop） | 通用动画 | 循环/分支完全没画出来，与内容深度最不匹配 | **需要专项重做**（环形 Loop，第一优先） |
| `issue-fix-flow`（Issue Fix Agent） | 通用动画 | 失败回退分支与质量回流没画 | **需要专项重做**（带回退箭头流程图） |
| （Skill 无动画） | 友好降级文案 | 干净、不误导 | **暂时 fallback 即可** |

结论：

- **可以保留**：KV Cache（小修即可）。
- **需要动画工程师专项优化**：其余 7 类。它们目前不是"能解释机制的动画"，而是"被包装成动画的 fallback"，且把内部 key 暴露给用户。
- **暂时 fallback 即可**：无动画的 Skill 维持降级文案。
- 重要判断：动画当前**不构成产品差异化能力**，但也**没有严重误导理解**（所以归 P1 而非 P0）；问题在于"解释力不足 + 内部标识泄漏 + 会被 44 讲批量复制"。在扩展前，至少应把 `prefill-decode` 与 `agent-loop` 两个高复用类型做成真实画布，否则样板会把"假动画"复制 30+ 次。

## 6. 诊断题体验评审

- **是否像真实工程题**：像。全部是"现象 + 指标 + 让你选优先动作/排查方向"的场景题，例如"账单涨 180% 但输出长度没变→先查输入 Token 膨胀"、"TTFT 涨而 TPOT 稳→瓶颈在 Prefill"。这是工程判断题，不是名词解释。
- **是否能训练判断力**：能。解析普遍给出"为什么这个优先、为什么别的虽然有用但不该先做"的工程理由，并配"推荐排查路径"，这是本产品**最接近差异化能力**的部分。TTFT 多选题（A/B/C 对、D 错）质量尤其好。
- **是否太简单或太考试化**：偏套路化，存在两个可被利用的规律：
  1. 11 道单选里 **8 道正确答案是选项 B**（token/attention/decode/kv-cache/multi-model-routing/context-window/skill/agent-loop 均为 b）。聪明的应试者会形成"选 B"惯性。
  2. 正确答案几乎总是"先建立流程/评测/度量/约束"那一项，干扰项几乎总是"直接扩容/直接降温/直接换模型/直接加工具"。教学上正确，但太可预测，迷惑性来自"幼稚的快速修复"而非"两个都挺合理"。
- **最值得保留**：`q-ttft-1`（多选，链路拆分）、`q-kv-cache-1`（命中率+亲和）、`q-agent-loop-1`（可控/可追踪/可停止）、`q-issue-fix-agent-1`（先复现再最小修改）——都是好题。
- **最需要重写/调整**：`q-skill-1` 与 `q-model-gateway-1` 的干扰项偏弱（一眼能排除），可增强迷惑性；并打散正确答案位置、让至少 1–2 道的正确答案是"在两个都合理的工程动作里二选一"，而不是"对 vs 明显幼稚"。
- **诊断题能否成为差异化能力**：**能**，且应被当作核心卖点经营。前提是扩展 44 讲时把"答案分布 + 干扰项强度"纳入内容门禁，否则套路会被批量复制。

## 7. 学习路径与闭环体验

检查链路：首页 → 模块页 → 知识点详情 → 动画 → 诊断题 → 完成 → 我的学习 → 搜索。

- **是否顺畅**：单页内顺畅（详情页结构完整、动画可播放、诊断可提交、完成/收藏即时反馈、Profile 数据准确、刷新不丢、清空可用）。**页与页之间不顺畅**：
  - 详情页"下一个"按 56 讲全局顺序取下一条，**12 讲里有 9 讲的"下一个"会跳进 stub 空白页**（仅 prefill→decode、decode→ttft、model-gateway→multi-model-routing 落在正文上）。
  - 模块页列出该模块全部知识点，M1 显示 10 张卡但仅 2 张有正文，其余 8 张是"内容草稿待入库"，模块页观感像未完工。
  - 搜索命中按标题匹配，可命中 44 个 stub，点进去是空白页。
  - 没有一条贯穿 12 讲的"推荐下一讲"路径；唯一的顺序线索是模块顺序，而模块顺序里全是 stub。
- **是否有继续学习动力**：进度反馈（顶/侧/Profile 三处 done/total + 连续天数 + 完成绿色态）做得好，有动力来源；但每点一次"下一个"就可能掉进空房间，会快速消解动力。
- **是否能证明 56 讲产品形态成立**：**结构上成立，闭环演示上打折**。页面骨架、内容骨架、诊断题骨架、进度系统都证明了形态可扩展；但"自然点击就掉进 stub"让人无法干净地走完一条 12 讲闭环，**这正是当前最该在定稿前修掉的样板级问题**。
- **问题**：见 P0-01（导航走进 stub）、P1-01（模块页 stub 噪声）、P1-02（搜索 stub 噪声）。

## 8. 视觉风格一致性

对照 design.md：

- **符合项**（强）：
  - `tokens.css` 与 design.md §2 颜色/字体/尺寸逐项一致；背景暖纸色 `#FAF9F6`，主色蓝 `#1F40D8`，进度绿 `#2E7D58`。
  - serif 标题 / sans 正文 / mono 序号指标的三字体层级正确落地（global.css）。
  - 未用华为红、无大面积深色（深色仅限动画机制画布、面积克制）、无霓虹渐变/AI 光效、边框优先于阴影、圆角 6–8px。
  - 详情页正文宽度 860px、首页 1120px，长阅读舒适；信息密度低噪音。
  - 整体不像"AI 工程监控后台"、不像 Notion 皮肤、不像企业官网——像安静的教材。
- **偏离项**：
  - 详情页头部把 `contentStatus`（`mvp`）当 meta 标签直接展示给用户；design.md 的头部示例是"高级 · 12 分钟 · 含动画演示"三项，没有成熟度标签（P1-03）。
  - 通用动画把 `config.type`（如 `token-flow`）和 `highlightTargets` 英文 key 直接铺成标签，是把内部标识当视觉元素，破坏中文教材质感（P1-05）。
- **最影响产品质感的问题**：上述两类"内部英文标识泄漏到阅读界面"。它们单点看很小，但会随 56 讲被复制到每一页/每个动画。
- **建议**：详情页头部移除/改写成熟度标签为面向用户的语义；通用动画在被专用画布替换前，至少不要把内部 key 暴露给用户。

## 9. P0 必须修复项

> P0 定义：不修复就不建议扩展 44 讲（或不修复就无法干净演示 12 讲闭环）。

### P0-01：课程内导航频繁把用户送入空白 stub 页，12 讲闭环在自然点击下是"漏的"

- 问题：详情页"下一个"按全局 56 讲顺序取下一条（`ConceptPage.tsx` 的 `orderedConcepts[index+1]`），12 讲中 9 讲的"下一个"落在 stub；首页"继续学习"在用户访问过 stub 后也会回到 stub；模块页/搜索同样可直接进入 stub 空白页。
- 影响：用户/评审顺着产品自然点击，会反复掉进"内容草稿待入库"的空房间，无法干净走完一条 12 讲学习闭环——这直接削弱"证明 56 讲形态成立"的样板价值。
- 涉及页面 / 文件：`src/pages/ConceptPage.tsx`（下一个）、`src/utils/progress.ts`（继续学习目标）、`src/pages/ModulePage.tsx`、`src/pages/SearchPage.tsx` / `src/utils/search.ts`。
- 建议修复（按 MVP 阶段，最小改动）：让"下一个/继续学习/推荐"在当前阶段优先在**有正文的知识点（contentStatus ∈ {demo,mvp}）**之间流转（例如"下一讲"跳到下一条有正文的；或对 stub 卡片/搜索结果加"待入库"弱化态并禁跳）。不改 schema、不改内容，纯前端导航策略。
- 是否需要 Owner 确认：需要。决策点是"MVP 演示期是否隐藏/弱化 44 个 stub 入口"还是"保留 stub 入口但加明显占位态"——两者都能解 P0，方向需 Owner 拍板（见第 12 节）。

## 10. P1 建议修复项

> P1 定义：影响样板质量，会被 44 讲批量复制，但不一定阻塞扩展启动。

### P1-01：模块页被 stub 卡片淹没

- 问题：模块详情页列出该模块全部知识点，M1/M4 等模块里大多数卡片是"内容草稿待入库"，模块页观感像半成品。
- 建议：MVP 阶段对 stub 卡片做弱化占位态（置灰、标"即将上线"、可不可点由 Owner 定），或提供"只看已上线"筛选默认值。

### P1-02：搜索命中 stub，点进去是空白页

- 问题：`searchConcepts` 按标题/标签命中，可命中 44 个 stub；结果项只显示"内容草稿待入库"，跳转后仍是空白详情。
- 建议：搜索结果对 stub 标注"待入库"并降权或在 MVP 期过滤；与 P0-01 同一套"已上线优先"策略。

### P1-03：详情页头部泄漏 `contentStatus`（mvp/stub）给用户

- 问题：`ConceptHeader.tsx` 直接渲染 `{concept.contentStatus}`，用户看到英文 `mvp`；偏离 design.md 头部示例。
- 建议：移除该标签，或转为面向用户的语义（如难度/是否含动画已足够）。

### P1-04：`kv-cache` 存在重复数据（内联 demo 版被 mvp 版覆盖成死数据）

- 问题：`src/data/concepts.ts` 内联保留了一份 `kv-cache`（`contentStatus:'demo'`），`demoConcepts.ts` 又有一份 `mvp` 版，合并逻辑 `demoById.get(id) ?? concept` 使内联版永远不生效。
- 影响：维护陷阱，扩展时易误改无效副本；也是数据所有权（内容只应在数据层单一来源）的轻微回归。
- 建议：删除 `concepts.ts` 内联的 `kv-cache` 长对象，统一改回 `stub(...)` 登记，正文只留在 `demoConcepts.ts`。

### P1-05：通用动画把内部英文 key 暴露为 UI 标签 + 步说明重复展示

- 问题：`GenericMechanismAnimation` 渲染 `config.type` 文本与 `highlightTargets`（`input-text / human-review / shifted-weights`…）英文 key；同一步标题/描述在画布内与播放器 caption 出现两次。
- 影响：中文教材里出现一排调试感英文标识，质感下降；7 个动画类型会把这一模式复制到 30+ 讲。
- 建议：在专用画布落地前，通用画布不展示 `type` 与原始 key，去掉重复 caption；专用画布把 key 映射为可视元素而非文本。

### P1-06：诊断题答案分布与干扰项强度套路化

- 问题：11 道单选 8 道答案为 B；正确项总是"先建流程/度量"，干扰项总是"幼稚快速修复"。
- 建议：打散答案位置；为部分题设计"两个都合理、择优"的强干扰项；把"答案分布 + 干扰项强度"写入内容生产检查清单，避免 44 讲复制套路。

## 11. P2 后续优化项

> P2 定义：可后续增强，不影响当前样板判断。

- **P2-01 内容模板化痕迹**：机制多为 6 条、结论恒 5 条、心智模型恒以"可以把 X 理解为…"开头，长期连读偏工整单调；可在写作清单里鼓励结构与句式适度变化。
- **P2-02 Google Fonts 外链**：`index.html` 直引 fonts.googleapis.com，受限网络 console 报错、字体回退（已在封版报告记录为 P1-01）；可本地化 IBM Plex 或接受系统字体。建议演示前处理。
- **P2-03 缺少"上一个"与显式 12 讲学习路径**：详情页只有"下一个"，且无一条贯穿 12 讲的推荐主线；可增加"上一讲/下一讲（已上线）"与一条策展路径。
- **P2-04 reduced-motion 下播放按钮禁用**：尊重系统设置正确，但可补充提示文案说明"已为你关闭自动播放，可手动逐步"。
- **P2-05 KV Cache 动画小修**：配色与 reduced-motion 静态态可再打磨，对齐到未来统一的动画视觉规范。

## 12. 是否可以进入下一阶段

- **是否可以开始扩展剩余 44 讲**：**有条件可以**。内容/页面/诊断题/进度的样板已足够稳定可复制；但建议**先修 P0-01（成本很低，纯前端导航策略）**，再开始扩展，否则每新增一讲都在加重"点进去是空房间"的问题。
- **是否需要先修 P0**：**是**，P0-01 必修（或至少先确定 Owner 的 stub 展示策略）。
- **是否需要先修 P1**：建议把 P1-03、P1-05、P1-04 在"定稿样板"时一并修掉，因为它们会被 44 讲**批量复制**（成熟度标签、动画 key 泄漏、数据重复模式）。P1-01/02/06 可与扩展并行。
- **是否需要先让动画工程师专项介入**：**是，但不必阻塞扩展启动**。底线是在大规模复用 `prefill-decode` 和 `agent-loop` 之前，先把这两个高复用类型做成真实画布；其余可滚动推进。当前 7/8 动画是"包装过的 fallback"。
- **是否需要先让内容 Agent 修订 12 讲**：**不需要大修**。12 讲正文质量达标，可直接作为写作样板；内容侧只需做 P1-06（诊断题答案分布/干扰项）和 P2-01（模板化）的轻量打磨，并把这两条写进内容生产检查清单。

### 给内容 Agent（只改内容结构、案例、诊断题）
