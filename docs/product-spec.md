# 产品规格 · product-spec

> 来源：PRD（产品需求）+ 56 讲内容开发素材（信息架构）+ design.md（信息密度与视觉）。
> 三条权威决策见 [content-schema.md](content-schema.md) 与 [AGENTS.md](../AGENTS.md)。

## 1. 产品定位

面向企业 AI 应用负责人、MaaS / AI Infra / Agent 平台负责人、研发管理者与资深工程师的**交互式学习系统**。不是 AI 百科，也不是提示词技巧合集，而是一套企业级 AI 应用工程教材。

核心目标：把关键知识点从“听过概念”提升到**能解释机制、能判断方案、能诊断问题、能指导落地**。

## 2. 目标用户

- 企业 AI 应用负责人 / AI 转型项目 Owner
- MaaS / AI Infra / Agent 平台负责人
- 企业研发管理者、资深软件工程师
- 想从“会用 AI 工具”升级到“能设计 AI 系统”的技术人员

用户已有软件工程基础，关心的是“为什么有效、怎么落地、出问题怎么定位、价值怎么评价”，而非“概念是什么”。

## 3. 首版范围

### 3.1 必须做

首页、模块列表页、知识点详情页、动画演示组件、诊断题组件、进度记录、收藏、搜索、响应式布局。术语索引页（GlossaryPage）首版做基础版本（见 §4.7）。

### 3.2 不做（硬边界）

后端 / 登录注册 / 付费 / 复杂后台 / 多人协作 / 真实 AI 问答 / 大模型 API / 原生端 / 视频播放器 / 完整题库系统。

### 3.3 首版分期：MVP Demo 与 MVP 1.0

首版偏满，拆成两个可交付节点，降低短周期演示风险。两期都要求 56 讲信息架构完整（登记齐全），区别在“完整内容”的体量与功能面。

| 维度 | MVP Demo（先达成，可演示） | MVP 1.0（再补齐） |
|---|---|---|
| 完整知识点 | 3–5 讲（建议 Token、KV Cache、Agent Loop + 1–2 讲） | ≥ 12 讲 |
| 诊断题 | 2–3 道 | ≥ 8 道 |
| 动画 | 2 类（建议 token-flow、kv-cache） | ≥ 4 类、≥ 6 个动画配置 |
| 页面 | 首页、模块页、详情页、动画、诊断题（核心闭环） | 补齐搜索、术语、Profile、响应式打磨 |
| 门禁 | `validate:content` 通过 | `validate:content` 通过 + 验收清单全绿 |

> Demo 用于跑通内容结构、动画风格与诊断题质量；1.0 形成完整主线支撑 MVP 演示。

### 3.4 内容门槛（MVP 1.0 目标）

- 56 讲信息架构完整（6 模块、56 知识点登记齐全）。
- 至少内置 **12 讲**完整内容（“完整”判定见 [content-schema.md](content-schema.md) §6.2）。
- 至少 **8 个**知识点含诊断题。
- 至少 **6 个**知识点含动画配置，覆盖至少 **4 类**动画。

### 3.5 PWA 范围

- 首版（Demo 与 1.0）只做**静态 Web，预留 PWA**：可提供 `manifest.webmanifest`（名称、图标、主题色）实现基础可安装性（basic installability），**但不做 Service Worker、不做离线缓存策略**。
- 明确不在首版做的：离线可用、后台同步、资源预缓存、安装引导浮层。
- 即“静态 Web + 可选 manifest”；完整 PWA（离线/缓存）列为二期，避免过早实现 Service Worker。

## 4. 页面规格

每个页面遵循 design.md 的信息密度规则：第一屏只放当前任务最需要的信息。

### 4.1 首页（`/`）

- 目标：让用户立刻知道这是什么产品，并能继续学习。
- 第一屏只放：左侧导航、顶部搜索、连续学习天数、主标题区、继续学习按钮、简洁学习进度、推荐学习路径露出一小部分。
- 核心动作：**继续学习**（进入上次学习位置 / 第一个未完成知识点）。
- 其余区块（六大模块卡片、今日推荐）置于页面下方，不挤进第一屏。
- 交互：点模块进专题列表；点知识点进详情页；点搜索开搜索；点“查看我的学习”进 Profile。

### 4.2 模块列表页（`/modules/:moduleId`）

- 展示某模块下全部知识点及学习路径说明。
- 知识点卡片字段：标题、一句话定义、难度、预计时长、是否有动画、是否完成、是否收藏。
- 交互：进详情页、切换收藏、按难度/状态/是否有动画筛选、按推荐/时长/难度排序。

### 4.3 知识点详情页（`/concepts/:slug`）

- 目标：把一个知识点从概念、机制、案例、诊断题四层吃透；进入长阅读状态。
- 页面结构（顺序固定）：
  1. 标题区（难度 · 时长 · 是否含动画）
  2. 一句话定义（左侧 3px 蓝色竖线强调）
  3. 为什么重要
  4. 心智模型（浅暖灰底块）
  5. 机制讲解（编号步骤，mono + 蓝色序号）
  6. 动画演示（若有，见 [animation-spec.md](animation-spec.md)）
  7. 企业案例（场景/问题/分析/方案/结论）
  8. 常见误区
  9. 诊断题（若有）
  10. 核心结论
  11. 关联知识点
  12. 完成学习 / 收藏 / 下一个
- 正文最大宽度约 760–860px，保持长阅读节奏，不拆成过多卡片。

### 4.4 动画演示组件

详见 [animation-spec.md](animation-spec.md)。要点：作为详情页中的一个章节，不是独立控制台；允许深色机制画布但面积克制；控制区仅上一步/播放暂停/下一步/重置/步骤计数；每次只解释当前一步。

### 4.5 诊断题组件

- 目标：用企业真实场景训练工程判断，不考定义。
- 结构：场景描述、现象、可选判断、用户选择、解析、推荐排查路径、关联知识点。
- 题型：首版只做单选 / 多选。
- 行为：提交后判定对错、展示解析与排查路径，错题写入错题本。

### 4.6 搜索页 / 浮层（`/search`）

- 范围：标题、一句话定义、机制讲解、企业案例、常见误区、诊断题（首版至少覆盖 title / definition / tags / mechanism / enterpriseCase / pitfalls）。
- 交互：实时搜索、结果可点击跳转、空结果提示、支持 `/` 快捷键唤起、`esc` 关闭。
- 排序：标题完全匹配 > 标题包含 > 标签匹配 > 正文匹配。

### 4.7 术语索引页（`/glossary`）

- 内容：每个术语含中文名、英文名、一句话解释、所属模块、关联知识点。
- 首版做基础列表版本；高级筛选/字典化排版列为二期。

### 4.8 我的学习页（`/profile`）

- 内容：总完成进度、各模块完成进度、最近学习、收藏知识点、错题记录、推荐复习、清空学习记录。

## 5. 信息架构（56 讲权威清单）

模块构成 `10 / 10 / 8 / 16 / 6 / 6`，共 **56** 个知识点。逐条 id/slug/难度/时长/动画登记见 [content-schema.md](content-schema.md) §4。

| 模块 | 标题 | 讲数 | 目标 |
|---|---|---|---|
| 01 | 模型怎么工作 | 10 | 理解大模型基本机制与能力边界 |
| 02 | 模型怎么跑得又快又稳 | 10 | 理解推理性能、时延、吞吐、缓存、调度、瓶颈 |
| 03 | 模型怎么变成企业平台 | 8 | 理解 MaaS、网关、路由、缓存、成本、SLA |
| 04 | 模型怎么变成 Agent | 16 | 理解上下文工程、Agent Loop、工具、Skill、多 Agent |
| 05 | Agent 怎么改变软件工程 | 6 | 理解 AI 原生软件工程闭环 |
| 06 | 企业怎么治理 AI | 6 | 理解评测、观测、安全、权限、成本、组织 |

### 推荐学习路径（首页露出 4 条）

- 入门路径：模型怎么工作（M1）
- 工程路径：推理系统与 MaaS（M2）
- 应用路径：Agent、上下文与 Skill（M4）
- 负责人路径：成本、评测、治理与组织（M6）

## 7. MVP 优先级

### 页面落地顺序

1. 首页 → 2. 模块列表页 → 3. 知识点详情页 → 4. 动画演示组件 → 5. 诊断题组件 → 6. 搜索页 → 7. 我的学习页

### 视觉优先级

1. 先保证首页和知识点详情页气质正确（暖纸 + serif 教材感）。
2. 再保证动画组件清晰。
3. 最后补齐诊断题、搜索与进度统计。

### 成功标准

用户打开即理解这是 AI 应用工程学习系统；能沿 6 模块形成知识地图；能通过动画看懂机制、通过诊断题训练判断；进度可记录；结构清晰可扩展到 100 讲；内容/动画/题目都由数据配置驱动。
