# AI 工程负责人日常学习增强 PRD

> 版本：v0.1  
> 日期：2026-06-24  
> 适用基线：56 / 56 讲已上线、正文 v2、Electron 桌面版 MVP 已完成后的下一阶段增强。  
> 产品定位延续：[product-spec.md](product-spec.md)、[content-schema.md](content-schema.md)、[AGENTS.md](../AGENTS.md)。本 PRD 只定义增强方向与验收口径，不直接修改现有内容 schema。

## 1. 背景

当前 APP 已完成 56 讲 AI 应用工程知识体系，适合系统学习概念、机制、案例与诊断题。对大型企业 AI 工程负责人而言，下一阶段的核心价值不再是继续堆课程数量，而是把已有知识转化为日常工作中的三类能力：

1. 做技术方案判断。
2. 诊断复杂工程问题。
3. 识别个人与团队的能力缺口。

因此，下一阶段应从“交互式课程 APP”升级为“AI 工程负责人日常决策与复盘工具”。

## 2. 目标用户

核心用户：

- 大型企业 AI 应用工程负责人。
- MaaS / AI Infra / Agent 平台负责人。
- 负责 AI 应用落地、治理、成本、质量与团队能力建设的技术管理者。

典型使用场景：

- 评审一个 RAG、Agent、模型路由、MaaS 网关或企业治理方案前，快速查判断框架。
- 生产问题复盘时，按现象定位可能原因和排查顺序。
- 每周安排自己或团队的 AI 工程学习重点。
- 给业务、管理层或平台团队解释某个技术取舍。

## 3. 产品目标

### 3.1 业务目标

- 让用户从“学过知识点”进一步变成“能拿知识点做工程决策”。
- 把诊断题从单讲练习升级为跨知识点的真实场景演练。
- 让 Profile 从学习进度页升级为负责人视角的能力地图。

### 3.2 用户目标

用户使用增强版后，应能完成：

- 快速判断某个 AI 工程方案是否适合当前企业场景。
- 遇到成本、延迟、质量、安全、治理类问题时，有可执行的排查路径。
- 看清自己和团队在 AI 应用工程能力上的短板。
- 把学习成果转化为评审清单、复盘清单和团队学习路径。

### 3.3 非目标

本阶段不做：

- 后端、账号、云同步、多人协作。
- 真实大模型 API 调用。
- 企业内部系统集成。
- 真实团队成员管理。
- 完整题库平台或内容后台。

团队相关能力仅做“角色路径 / 能力模型 / 本地复盘视图”，不做在线协作。

## 4. Top 3 增强方向

### 4.1 方向一：AI 工程决策手册化

### 4.1.1 问题

当前每讲已经能解释概念、机制和案例，但企业负责人在真实工作中更常问：

- 这个方案什么时候该用？
- 什么情况下不该用？
- 代价是什么？
- 评审时该问哪些问题？
- 风险边界在哪里？
- 如何向管理层解释这个取舍？

这些内容现在分散在定义、机制、案例、误区和诊断题中，尚未形成负责人可直接使用的决策结构。

### 4.1.2 目标

为重点知识点补充“决策手册层”，让每讲成为一个可复用的方案评审单元。

优先覆盖：

- RAG 与上下文工程。
- 模型路由、KV Cache、推理性能、Token 成本。
- Tool Calling、Agent Loop、多 Agent。
- Trace / Observability / Eval / Guardrails。
- 权限、安全、Session 亲和、成本治理。

### 4.1.3 功能需求

每个重点知识点新增一个“工程决策”区域，建议包含：

| 子模块 | 说明 |
|---|---|
| 适用场景 | 哪些业务、系统规模、组织条件下适合采用该方案。 |
| 不适用场景 | 哪些情况下不要优先采用，避免方案误用。 |
| 决策信号 | 做选择前必须确认的指标、约束或系统事实。 |
| 架构取舍 | 成本、延迟、质量、可观测性、安全、运维复杂度之间的权衡。 |
| 评审问题 | 技术评审会上负责人应追问的问题。 |
| 落地清单 | 实施前、上线前、运行期要检查的事项。 |
| 管理层解释 | 用非细节化语言说明为什么这样选、风险是什么、怎么控。 |

### 4.1.4 页面体验

知识点详情页新增一个“工程决策”章节，位置建议在“企业案例”之后、“常见误区”之前。

交互要求：

- 默认展开前 3 个最高价值判断项。
- 支持复制评审清单。
- 支持将该讲加入“本周复盘”。
- 不使用复杂卡片堆叠，保持现有长文阅读节奏。

### 4.1.5 内容生产要求

每条决策内容必须满足：

- 绑定真实工程约束，例如规模、延迟、成本、权限、评估、运维边界。
- 避免泛泛写“视情况而定”，必须说明判断信号。
- 至少给出 3 条评审问题。
- 至少给出 3 条落地检查项。

如需落库，应先在 `docs/content-schema.md` 提出 schema 变更，再同步更新 `src/types/index.ts` 与 `validate:content`。

### 4.1.6 验收标准

- 首批至少覆盖 12 个高频负责人决策知识点。
- 每个覆盖知识点均包含适用场景、不适用场景、评审问题、落地清单。
- 至少 80% 的评审问题能直接用于一次技术方案评审。
- `validate:content`、`typecheck`、`lint`、`build` 通过。

### 4.2 方向二：真实场景诊断演练

### 4.2.1 问题

现有诊断题已经训练单讲判断，但企业负责人面对的问题通常是跨模块的：

- 成本突然上涨，可能来自模型选择、上下文、缓存、重试、工具调用或流量结构。
- RAG 答案质量差，可能来自召回、重排、上下文注入、评估集或权限过滤。
- Agent 卡死，可能来自循环控制、工具协议、状态管理、观察信号或权限失败。

单讲题目无法充分训练跨知识点定位能力。

### 4.2.2 目标

新增“场景演练”能力，让用户在接近真实生产问题的画布中做判断、排查和复盘。

优先实现一个标杆场景：`model-router` 真实画布升级。

### 4.2.3 场景库规划

首批建议 6 个场景：

| 场景 | 训练目标 | 关联模块 |
|---|---|---|
| 模型路由策略失效 | 在成本、延迟、质量之间调策略 | M2 / M3 / M6 |
| RAG 召回正常但答案差 | 区分召回、重排、上下文与评估问题 | M4 / M6 |
| Agent 工具调用失败 | 判断协议、权限、参数、重试与状态问题 | M4 / M6 |
| Token 成本异常上涨 | 定位上下文膨胀、重试、缓存缺失、模型误选 | M2 / M3 / M6 |
| Trace 有数据但不可诊断 | 区分观测字段、链路边界与敏感数据最小化 | M6 |
| 多 Agent 流程卡住 | 判断编排、角色边界、共享状态与终止条件 | M4 / M5 |

### 4.2.4 标杆功能：模型路由真实画布

页面形态：

- 作为模型路由相关知识点的综合动画 / 演练升级，对应 `multi-model-routing`、`cost-routing`、`capability-routing`。
- 不接真实模型 API，使用本地模拟数据。
- 用户通过调节策略观察成本、延迟、成功率、质量风险的变化。

核心元素：

| 元素 | 说明 |
|---|---|
| 请求队列 | 展示不同任务类型：简单问答、长文总结、代码生成、合规高风险请求。 |
| 路由策略 | 支持按任务类型、风险等级、上下文长度、SLA 选择模型。 |
| 模型池 | 展示快模型、强模型、低成本模型、受限模型的模拟指标。 |
| 指标面板 | 成本、P95 延迟、成功率、升级率、风险拦截率。 |
| 诊断事件 | 展示“质量下降”“成本上升”“SLA 违约”等异常。 |
| 复盘结论 | 用户提交判断后，给出原因、排查顺序和关联知识点。 |

### 4.2.5 交互流程

1. 用户进入场景，看到业务背景和初始指标。
2. 系统给出异常现象，例如“成本下降但投诉升高”。
3. 用户查看请求分布、路由规则和模型池。
4. 用户调整 1 到 3 个策略项。
5. 系统实时更新模拟指标。
6. 用户提交诊断结论。
7. 系统展示正确排查路径、遗漏风险和关联知识点。

### 4.2.6 数据与实现约束

- 所有场景数据必须在本地 `src/data/*` 或配置文件中维护。
- 模拟指标必须可解释，不追求真实数学模型复杂度。
- 不引入后端，不调用外部 API。
- 场景状态仍走现有 Zustand 进度体系或本地组件状态；如需持久化，必须扩展 `progressStore.ts`。

### 4.2.7 验收标准

- 首批至少实现 1 个完整场景演练，建议为 `model-router`。
- 场景至少包含 3 个可调策略项和 4 个可观察指标。
- 用户每次策略调整后，指标变化必须有清晰解释。
- 提交诊断后必须展示排查路径和关联知识点。
- 场景在桌面和移动端均可用，不出现关键控件溢出。

### 4.3 方向三：个人 / 团队学习驾驶舱

### 4.3.1 问题

现有 Profile 更接近进度页，能告诉用户学了多少，但不能回答负责人更关心的问题：

- 我在哪些 AI 工程能力域薄弱？
- 最近错题反映出什么判断偏差？
- 下周该优先复习什么？
- 如果我要带团队补能力，应该按什么角色路径安排？

### 4.3.2 目标

把 Profile 升级为“学习驾驶舱”，用本地学习行为、完成情况、错题和收藏生成能力画像与推荐路径。

### 4.3.3 能力模型

建议将 56 讲映射到 7 个负责人能力域：

| 能力域 | 说明 |
|---|---|
| 模型机制理解 | Token、注意力、上下文、能力边界。 |
| 推理性能与成本 | 延迟、吞吐、KV Cache、批处理、成本优化。 |
| MaaS 与平台化 | 网关、路由、SLA、配额、缓存、平台治理。 |
| RAG 与上下文工程 | 检索、重排、上下文构造、知识库质量。 |
| Agent 工程 | Agent Loop、Tool Calling、状态、权限、多 Agent。 |
| 评估与可观测 | Eval、Trace、指标、复盘、质量回归。 |
| 安全治理与组织落地 | 权限、数据边界、Guardrails、成本治理、组织机制。 |

每个知识点可映射 1 到 2 个能力域。

### 4.3.4 功能需求

Profile 页面新增：

| 模块 | 说明 |
|---|---|
| 能力雷达 / 条形图 | 展示 7 个能力域的学习进度与诊断表现。 |
| 本周建议 | 根据未完成、错题、收藏生成 3 个学习建议。 |
| 判断偏差 | 汇总最近错题类型，例如“优先看工具而忽略权限边界”。 |
| 复盘清单 | 汇总最近完成的讲、错题、收藏和需要复习的概念。 |
| 角色路径 | 提供负责人、平台工程师、应用架构师、治理负责人 4 条路径。 |
| 下一个行动 | 明确给出下一讲、下一题或下一场景演练。 |

### 4.3.5 搜索 / 术语联动

搜索页增强：

- 支持按能力域过滤。
- 支持搜索“我要排查成本上涨”“我要评审 RAG 方案”等意图型关键词。
- 搜索结果显示对应的决策手册、场景演练和术语。

术语页增强：

- 术语关联能力域。
- 术语页展示“常被混淆概念”。
- 从术语可跳转到对应决策章节或场景演练。

### 4.3.6 本地团队路径

本阶段不做真实团队管理，只提供“角色学习路径模板”：

| 角色 | 重点 |
|---|---|
| AI 工程负责人 | 成本、质量、治理、平台化、组织落地。 |
| 平台工程师 | MaaS、路由、缓存、SLA、观测。 |
| 应用架构师 | RAG、Agent、上下文工程、工具调用。 |
| 治理负责人 | 权限、安全、评估、Trace、合规边界。 |

用户可以查看某个角色路径的推荐讲次、建议顺序和已完成比例。

### 4.3.7 验收标准

- Profile 至少展示 7 个能力域中的完成度。
- 推荐逻辑至少同时考虑未完成、错题、收藏中的两类信号。
- 至少提供 4 条角色路径模板。
- 搜索结果可按能力域过滤。
- 不引入后端、账号或团队数据结构。

## 7. 推荐分期

### Phase 1：负责人决策 MVP

目标：先证明“课程内容能变成决策工具”。

范围：

- 12 个重点知识点补“工程决策”章节。
- Profile 增加 7 个能力域完成度。
- Search 支持能力域过滤。

验收：

- 用户能从首页或 Profile 快速进入一个待复盘知识点。
- 知识点详情页能直接复制评审问题或落地清单。

### Phase 2：模型路由场景演练

目标：做出一个高质量标杆场景，建立后续场景模式。

范围：

- `model-router` 真实画布升级。
- 本地模拟请求、策略、模型池和指标。
- 提交诊断后展示复盘路径。

验收：

- 用户能通过调整策略理解成本、延迟、质量和风险的取舍。
- 场景能作为后续 RAG、Agent、Trace 场景的组件参考。

### Phase 3：学习驾驶舱完整化

目标：把个人学习、错题、收藏、角色路径整合为日常复盘入口。

范围：

- 本周建议。
- 判断偏差。
- 复盘清单。
- 4 条角色路径模板。
- Glossary 混淆概念增强。

验收：

- 用户打开 Profile 即可知道下一步学什么、为什么学。
- 用户能按角色路径安排一周学习计划。

## 8. 成功指标

本地产品可用以下指标衡量增强是否有效：

| 指标 | 目标 |
|---|---|
| 决策章节覆盖 | 首批 >= 12 讲，后续逐步覆盖 56 讲。 |
| 场景完成率 | 用户进入场景后完成提交的比例 >= 60%。 |
| 复盘入口使用 | Profile 中“下一步行动”点击率高于普通模块列表点击率。 |
| 搜索命中质量 | 高频意图词能返回相关知识点、决策章节或场景。 |
| 内容可复用性 | 评审问题和落地清单能直接用于真实方案评审。 |

## 9. 风险与对策

| 风险 | 对策 |
|---|---|
| 决策内容变成空泛管理话术 | 每条内容必须绑定工程信号、指标、边界或约束。 |
| 场景演练过度复杂 | 首批只做一个标杆场景，指标模拟保持可解释。 |
| Profile 变成拥挤 dashboard | 保持现有设计语言，第一屏只放能力概览和下一步行动。 |
| 团队路径被误解为多人协作 | 明确本阶段只做本地角色模板，不做账号和团队数据。 |
| schema 扩展破坏现有内容 | 先写 schema 变更，再更新类型、校验和数据。 |

## 10. 开放问题

1. 决策手册首批 12 讲的准确名单是否由 Owner 指定，还是按使用频率和负责人价值默认选取？
2. `model-router` 场景是否作为 `animation.type` 的增强，还是独立 `scenarioExercise` 类型？
3. 能力域得分是否只按完成度计算，还是同时计入诊断题表现？
4. Profile 是否需要导出本周复盘清单为 Markdown？
5. 角色路径是否允许用户本地自定义，还是先固定模板？
